What is this project?

The Big Picture

PoliMillionaire is an AI bot that plays "Who Wants to Be a Millionaire?" — a quiz show where 15 multiple-choice questions with increasing difficulty stand between you and €1,000,000. The bot connects to a live game server, receives questions across 4 categories (Entertainment, History, Science, Maths), and uses a suite of AI strategies to pick the best answer within a 30-second time limit.

How the System Works — In One Paragraph

The bot logs in to the game server, starts a game session, and enters a loop. For each question, it builds a StrategyInput (question text, 4 options, difficulty level, category) and hands it to the TieredStrategy. Depending on difficulty, the strategy either calls the bare LLM (easy), retrieves relevant Wikipedia passages via RAG (medium), runs a full ensemble (hard), or uses a deterministic calculator (maths). The chosen answer is submitted to the server, which returns correct/wrong. If a live Wikipedia search was used and the answer was correct, that article is permanently added to the knowledge index for future games.

Section 1

Three Layers — Why Not Just One?

Imagine a phone interpreter at a diplomatic summit. The diplomat (Strategy) speaks only English and doesn't know foreign protocol. The interpreter (Adapter) translates. The phone (HTTP client) handles transmission. Change any layer without touching the others.

LayerCodeResponsibility
1 — HTTPmillionaire_client/Raw HTTP calls, auth cookies, typed exceptions
2 — Adapterpolimibot/game/adapter.pyTranslates API types → domain types; fixes server quirks
3 — Runnerpolimibot/runner.pyGame loop, watchdog, rate limiter, logger, learning loop

The rule: Nothing outside adapter.py may import millionaire_client. All 30+ strategies, the evaluator, the logger — none ever touch the HTTP layer directly. If the server API changes, fix only adapter.py.

The Adapter — Type Translation

polimibot/game/adapter.py — key translations
def _to_game_question(q) -> Optional[GameQuestion]:
    """API Question → our frozen DTO."""
    if q is None: return None
    return GameQuestion(
        text=q.text,
        options=tuple(opt.text for opt in q.options),  # texts only, no IDs
        level=q.level,
    )

class GameAdapter:
    @property
    def current_question(self) -> Optional[GameQuestion]:
        q = _to_game_question(self._session.current_question)
        if q and q.level == 0:
            # Server bug: level sometimes missing → inject from session
            q = GameQuestion(text=q.text, options=q.options,
                           level=self._session.current_level)
        return q

    def submit_answer(self, option_index: int) -> AnswerOutcome:
        # Convert 0-based index → server option ID
        option_id = self._session.current_question.options[option_index].id
        result = self._session.answer(option_id=option_id)
        return AnswerOutcome(correct=result.correct, ...)
  • Options become plain strings — strategies never need server-side option IDs.Keeps strategies clean. Server internals don't leak up.
  • Server sometimes omits the level field → defaults to 0 → all questions route to "easy tier." The adapter patches this from the session state.Silent bugs are the worst. One defensive check prevents completely wrong routing.
  • The runner gives the adapter a 0-based index (0,1,2,3). The adapter resolves the actual server ID. Translation in one place only.Open/Closed: the rest of the system never needs to change if the server changes option ID format.

The Game Loop

polimibot/runner.py — play_game() core loop
while not game.is_over:
    q = game.current_question
    # Budget: server time minus 2s safety margin, capped at 25s
    budget = min(game.time_remaining_seconds - 2.0, 25.0)

    inp = StrategyInput(question=q.text, options=q.options,
                        level=q.level, category=category,
                        time_budget_seconds=budget)

    # Run strategy under watchdog (kills on overrun)
    try:
        out = _call_with_deadline(strategy, inp, budget)
    except _DeadlineExceeded:
        out = None   # timed out → use fallback

    chosen_idx = (out.chosen_index
                  if out and not out.is_abstain
                  else fallback_index)   # default: 0 (option A)

    pacer.wait()                         # ≥1.5s between API calls
    outcome = game.submit_answer(chosen_idx)

    # Learning loop
    grower = getattr(strategy, "index_grower", None)
    if grower:
        if outcome.correct: grower.confirm(f"lvl_{q.level}")
        else:               grower.discard(f"lvl_{q.level}")

    log.log_question(QuestionRecord(...))
Section 4

Watchdog & Pacer

_call_with_deadline — daemon thread timeout
def _call_with_deadline(strategy, inp, deadline_seconds):
    box = {}
    def target():
        try: box["out"] = strategy.answer(inp)
        except BaseException as e: box["err"] = e

    th = threading.Thread(target=target, daemon=True)
    th.start()
    th.join(deadline_seconds)   # block for at most this many seconds
    if th.is_alive():
        raise _DeadlineExceeded(...)
    # Thread can't be killed safely (GPU would corrupt) — it runs on
    # in the background but its result is ignored
    return box["out"]

Why daemon threads? Python can't safely interrupt a thread running CUDA kernels. We don't kill it — we just stop waiting. Daemon=True ensures the thread is cleaned up when the process exits, preventing zombie GPU processes.

What is RAG?

Imagine taking a quiz with an open encyclopedia. Before answering, you flip to the relevant pages, skim the key paragraphs, then use your brain to pick the answer. RAG does this for the AI: Retrieve (find the right pages) + Augment (add them to the prompt) + Generate (LLM answers with evidence).

The difference — what the LLM sees
# WITHOUT RAG — LLM relies only on memorized training data
"Question: When did Italy annex Rome?\nA) 1861  B) 1870  C) 1882  D) 1895\nAnswer:"

# WITH RAG — LLM sees retrieved evidence first
"""Reference material:
[1] Capture of Rome (chunk 2, history)
The capture of Rome on September 20, 1870 completed Italian unification...

[2] Italian unification (chunk 4, history)
The Kingdom of Italy moved its capital to Rome in 1871...

Question: When did Italy annex Rome?
A) 1861  B) 1870  C) 1882  D) 1895
Answer:"""   # → LLM sees "1870" twice → P(B) = 0.91
Section 2

Chunking — The Art of Splitting

Like making flashcards from a textbook. Cut at sentence boundaries, not mid-sentence. A card saying "Germany surrendered on May 7," is useless without "1945." — the date is on the next card.

❌ Bad: fixed-size character split
Chunk 1: "...surrendered unconditionally on May 7,"
Chunk 2: "1945. The war in Europe was over..."
✅ Good: sentence-aware split (your chunker)
Chunk 1: "...surrendered unconditionally on May 7, 1945. The war in Europe was over."
Chunk 2: "Japan continued fighting until August 15, 1945, when Emperor Hirohito..."

The last 50 words of Chunk 1 are copied to the start of Chunk 2 as overlap — so a fact at a boundary appears in BOTH chunks.

Section 3

Embeddings — GPS for Meaning

GPS coordinates, but for meaning. Texts with similar meaning get similar vector "coordinates." "Who invented the telephone?" and "Who created the phone?" are almost at the same location. "What's the weather?" is on a different continent.

Section 4

Search: FAISS + BM25 + RRF

Two librarians working in parallel. Librarian A (FAISS) understands meaning: "flying machines" → finds Aviation History. Librarian B (BM25) matches keywords: finds anything containing exact words. Together, they cover each other's blind spots.

MethodStrengthWeakness
FAISS (dense)Understands synonyms: "phone" = "telephone"Misses rare proper nouns, exact strings
BM25 (sparse)Perfect exact-match for names and rare termsBlind to synonyms and paraphrase
RRF FusionConsensus winners rank highestAdds computation; adds 1 RRF step
Reciprocal Rank Fusion — scale-independent combination
# RRF(d) = Σ  1 / (k + rank_i(d))     k=60, 1-indexed ranks
# FAISS scores: [-1,+1]  vs  BM25 scores: [0,30+]  — incompatible scales!
# RRF ignores scores, uses only RANK POSITIONS → always comparable

# Example: "Physics article" ranked #2 by FAISS, #1 by BM25
rrf = 1/(60+2) + 1/(60+1) = 0.03252   # HIGH IN BOTH → consensus winner!
# Even though it was #1 in neither list alone.
Section 5

Score Gating & Live Search

Bad context is worse than no context. A passage about "Rome, Georgia (USA)" when the question is about "Rome, Italy" will actively mislead the LLM. The score gate rejects low-quality retrieval before it reaches the prompt.

When the offline retrieval score is below the threshold, LiveSearchFallback fires a real-time Wikipedia API query, chunks the result on-the-fly, and uses those passages instead. If the answer turns out correct → the article is permanently added to the index (via IndexGrower) for future games.

PathScore rangeGate threshold
Dense (cosine)[-1, +1]≥ 0.65
Hybrid RRF[0.01, 0.10]≥ 0.055
Rerankermodel-dependent≥ 0.75
Section 6

IndexGrower — Self-Learning Knowledge Base

A student who glues useful pages into their textbook after a successful test answer. But if the answer was wrong, they don't add those pages — maybe the article was misleading.

polimibot/rag/index_grower.py — 4-phase lifecycle
grower.buffer(article, question_id)  # Phase 1: Live search fetched an article
                                     # Chunks available THIS question only
# After server verdict:
grower.confirm(question_id)  # Phase 2a: Correct → embed + add to FAISS & BM25 forever
grower.discard(question_id)  # Phase 2b: Wrong  → silently drop, don't pollute index
# End of game:
grower.flush()               # Phase 3: Persist in-memory growth to disk files

Only the Runner calls confirm/discard — the strategy doesn't know whether the answer was correct (that comes from the server AFTER submission). The runner is the only component that sees both the strategy output AND the server verdict.

Section 1

The Contract — One Interface for All Strategies

A game show host doesn't care how contestants decide. Genius, cheat sheet, or coin flip — the interaction is always: host gives question → contestant picks A/B/C/D. The Strategy pattern is that contract. Every algorithm implements the same interface so the runner can swap them without knowing internals.

polimibot/strategies/base.py — the Strategy contract
@dataclass(frozen=True)
class StrategyInput:
    question: str
    options: tuple[str, ...]          # always 4; index 0..3 = A..D
    level: int                        # 1..15 difficulty
    category: Optional[Category] = None
    time_budget_seconds: float = 25.0

@dataclass(frozen=True)
class StrategyOutput:
    chosen_index: int                 # 0..3
    confidence: float = 0.25          # honest uncertainty
    is_abstain: bool = False          # True → runner uses fallback
    extras: dict = field(...)         # probs, margin, passages, etc.

class Strategy(ABC):
    @abstractmethod
    def answer(self, inp: StrategyInput) -> StrategyOutput: ...

Section 3

Ensemble — Wisdom of Crowds

Ask three experts, combine their votes — weighted by confidence. If the calculator says 99% for option B but the LLM says 50% for B, the ensemble correctly lets the calculator dominate. If all three say B, the combined confidence is very high.

polimibot/strategies/ensemble_strategy.py — weighted fusion
def _weighted_fusion(self, active):
    combined = {"A": 0.0, "B": 0.0, "C": 0.0, "D": 0.0}
    total_w = sum(w for _, w in active)
    for out, w in active:
        probs = _extract_probs(out)  # full distribution or soft one-hot
        norm_w = w / total_w
        for letter in "ABCD":
            combined[letter] += norm_w * probs[letter]
    best = max(combined, key=lambda l: combined[l])
    return StrategyOutput(chosen_index=ord(best)-ord("A"),
                          confidence=combined[best])

Section 3

parse_answer — Regex Cascade

Like a post office sorting machine with 6 conveyor belts. Packages (LLM text) go down the first belt — if it matches "Answer: B" → done. Didn't match → second belt tries "\boxed{B}". Third belt tries "the answer is B". Final belt tries any bare letter. Reliably handles all common output formats.

polimibot/prompts/templates.py — parse_answer cascade
def parse_answer(text: str) -> Optional[int]:
    """6-level cascade. Returns 0-based index or None."""
    # Level 1: boxed format  \boxed{B}
    # Level 2: Answer: / Answer is B
    # Level 3: "therefore B" / "choose B"
    # Level 4: Option B / Choice B
    # Level 5: (B) isolated in brackets
    # Level 6: bare letter A/B/C/D as last resort
    for pattern, extractor in _CASCADE:
        m = pattern.search(text)
        if m: return ord(extractor(m).upper()) - ord("A")
    return None   # couldn't parse → strategy abstains

Tools & Computation

Deterministic arithmetic for maths questions, a sandboxed calculator, and a ReAct agent that calls tools in a reasoning loop.

MathsToolsafe_eval AST SandboxReActAgentStrategy
Section 1

Why Tools? — The LLM's Blind Spot

A brilliant literature professor trying to do your taxes. They can discuss Dostoevsky for hours but will mess up "What is 15% of $47,200?" LLMs are language machines, not calculators. When you need 2+2=4 (not "approximately 4"), you need a tool.

Precision over Recall. The tool only answers when 100% sure. If it can't parse the expression, or the result doesn't match any option → abstain, let the LLM try. A wrong deterministic answer (99% confidence) damages calibration more than a wrong LLM guess.

Section 2

safe_eval — AST Sandbox

eval() is a security hole. The LLM's output feeds into this. eval("__import__('os').system('rm -rf /')") executes arbitrary code. The AST sandbox makes this physically impossible — only whitelisted node types pass.

polimibot/tools/calculator.py — whitelist approach
_BINARY_OPS = { ast.Add, ast.Sub, ast.Mult, ast.Div, ast.Pow, ast.Mod, ast.FloorDiv }
_ALLOWED = { "sqrt": math.sqrt, "factorial": math.factorial, "log": math.log,
             "sin": math.sin, "cos": math.cos, "pi": math.pi, "e": math.e, ... }

def _eval_node(node):
    if isinstance(node, ast.Constant):
        if isinstance(node.value, (int, float)): return node.value
        raise ValueError()                    # no strings!
    if isinstance(node, ast.BinOp):
        if type(node.op) not in _BINARY_OPS: raise ValueError()   # no bitwise!
        return OP_MAP[type(node.op)](_eval_node(node.left), _eval_node(node.right))
    if isinstance(node, ast.Call):
        if not isinstance(node.func, ast.Name): raise ValueError()  # no method()!
        fn = _ALLOWED.get(node.func.id)
        if fn is None: raise ValueError()    # not in whitelist!
        return fn(*[_eval_node(a) for a in node.args])
    raise ValueError(f"AST node {type(node).__name__} not allowed")  # deny default
  • The final raise ValueError is the security critical line. Every unknown AST node type is rejected. New Python features can't sneak through.Whitelist (allow only safe things) beats blacklist (try to block dangerous things).
  • ast.parse(expr, mode='eval') rejects statements like x=5; x+1. Only pure expressions allowed.Prevents variable assignment, imports, loops — anything beyond arithmetic.
Section 3

AgentStrategy — ReAct Loop

A contestant who whispers to a friend with a calculator. They read the question → think aloud → say "what's 2^10?" → friend says 1024 → they conclude "option B" and answer. The agent thinks (Reason) and calls tools (Act) in an iterative loop.

AgentStrategy — the 3-branch loop
for iteration in range(max_iterations + 1):
    if time.monotonic() >= deadline: break

    resp = llm.generate(messages, max_new_tokens=200, temperature=0.0)
    text = resp.text.strip()
    messages.append({"role": "assistant", "content": text})

    # Branch 1: LLM emitted CALL: calc(expr) → execute tool
    call = _extract_call(text)
    if call:
        tool_name, expression = call
        observation = safe_eval(expression)  # deterministic computation
        messages.append({"role": "user",
                          "content": f"Tool result: {observation}\nNow give your final answer."})
        continue   # loop again; LLM sees the result

    # Branch 2: LLM emitted "Answer: B" → done
    idx = parse_answer(text)
    if idx is not None:
        return StrategyOutput(chosen_index=idx, confidence=0.70)

    # Branch 3: LLM is rambling → nudge it
    messages.append({"role": "user", "content": "Please write: 'Answer: <letter>'."})

Evaluation System

How we measure, compare, and scientifically prove that each strategy improvement actually works — offline, reproducibly, without burning GPU hours.

Section 1

Why Evaluate Offline?

Like preparing for a quiz show with practice tests. You collect past questions with known answers (gold set), test every new study method on the same set, and measure objectively. Differences reflect your method — not luck of the draw.

ProblemSolution
Can't replay live game questionsMine past games → frozen gold set
15 questions/game too noisyAggregate 100+ gold items across many games
Server downtime blocks developmentEvaluate 100% offline — no network needed

The cardinal rule: Build the gold set ONCE from baseline runs, then FREEZE it. Never re-harvest mid-experiment — that silently grows your test set and invalidates all previous comparisons.

Section 2

The Gold Set — Mining Ground Truth

polimibot/eval/gold_set.py — two recovery methods
def harvest_gold_set(runs_dir):
    """Mine run logs for correct answers using two methods."""
    confirmed = {}
    wrong_indices = defaultdict(set)

    for fp in runs_dir.glob("*.jsonl"):
        for rec in load_jsonl(fp):
            key = f"{cid}|{question_text}"
            if rec["correct"] is True:
                # Method 1: Direct — we submitted B and it was correct
                confirmed[key] = GoldItem(correct_index=idx, ...)
            elif rec["correct"] is False:
                # Accumulate wrong answers for elimination
                wrong_indices[key].add(idx)

    # Method 2: Elimination — if we've seen 3 of 4 options wrong, 4th is gold
    for key, wrong in wrong_indices.items():
        if key not in confirmed and len(wrong) == 3:
            correct_idx = (set(range(4)) - wrong).pop()
            confirmed[key] = GoldItem(correct_index=correct_idx, ...)

The GoldSet class wraps items with a chainable API: .filter_category("maths"), .filter_level(max_level=5), .take_per_level(3, seed=0), .split(0.8, seed=42). Every method returns a NEW GoldSet — never mutates.

ECE matters for the Ensemble. EnsembleStrategy weights votes by confidence. If a strategy says "90% sure" but is only right 60% of the time, it incorrectly dominates the ensemble. Good calibration → better fusion decisions.
