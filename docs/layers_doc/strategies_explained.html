Strategy 1

The Strategy Pattern — The Contract Everyone Signs

🏗️ Foundation

Imagine a game show. The host doesn't care whether you're a genius, have a cheat-sheet, or are flipping a coin — the interaction is always the same: host asks question → contestant picks A/B/C/D → host reveals answer. The Strategy pattern is that contract. Every "contestant" (algorithm) must follow the same interface, so the game-runner can swap them out without caring how they decide.

In software design, this is called the Strategy Pattern — one of the classic Gang-of-Four patterns. You define a common interface, and each implementation fills it differently. The caller (runner) doesn't know or care which algorithm is inside.

How It Looks in the Code

polimibot/strategies/base.py
@dataclass(frozen=True)
class StrategyInput:
    """Everything a strategy is allowed to see when answering one question."""
    question: str
    options: tuple[str, ...]          # always 4; index 0..3 = letters A..D
    level: int                        # 1..15 (difficulty increases)
    max_level: int = 15
    category: Optional[Category] = None
    competition_id: Optional[int] = None
    time_budget_seconds: float = 25.0


@dataclass(frozen=True)
class StrategyOutput:
    """What the strategy returns."""
    chosen_index: int                 # 0..3 → A..D
    confidence: float = 0.25          # in [0, 1]; default = random chance
    rationale: Optional[str] = None   # for logging/debugging
    is_abstain: bool = False          # True if strategy refused to answer
    extras: dict[str, Any] = field(default_factory=dict)


class Strategy(ABC):
    """Abstract base. Subclass and implement answer()."""
    name: str = "strategy"

    def warm_up(self) -> None:
        """Optional: load models, build indices."""
        return None

    def shutdown(self) -> None:
        """Optional: release resources."""
        return None

    @abstractmethod
    def answer(self, inp: StrategyInput) -> StrategyOutput:
        """Pick one option. Must return within inp.time_budget_seconds."""
  • StrategyInput is a frozen dataclass — the strategy sees the question, the 4 options, the difficulty level, category, and time budget. "Frozen" means strategies can't cheat by modifying their input.Separation of concerns: input is read-only data, output is the decision.
  • StrategyOutput carries the answer (0–3), a confidence score, optional reasoning text, and an abstain flag. If the strategy can't answer, it sets is_abstain=True and the runner substitutes a fallback.Honest uncertainty reporting enables downstream ensemble logic.
  • Strategy is abstract — you can't instantiate it directly. You MUST implement answer(). The optional warm_up() and shutdown() give strategies a lifecycle hook for loading/releasing GPU models.The contract guarantees every strategy is plug-and-play.

Why frozen dataclasses? If a strategy could modify the input, it might accidentally corrupt data for other strategies in an ensemble. Immutability prevents an entire class of bugs.

Strategy 2

RandomStrategy — The Statistical Floor

🎲 Baseline

Think of it as the monkey test. If you put a monkey in front of a 4-option quiz, it'll get ~25% right by chance. This strategy IS that monkey — it picks randomly. Every other strategy must beat 25% to prove it's doing anything useful. This is your "control group" in the experiment.

How It Looks in the Code

polimibot/strategies/random_pick.py
class RandomStrategy(Strategy):
    """Uniform random over the four options."""

    def __init__(self, seed: int | None = None) -> None:
        self.name = "random"
        self._rng = random.Random(seed)  # local instance, not global

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        n = len(inp.options)
        idx = self._rng.randrange(n)
        return StrategyOutput(
            chosen_index=idx,
            confidence=1.0 / n,   # honest: 25% confidence
            rationale=None,
        )
  • Uses a local random.Random(seed) instance — never touches the global random state. This means other code (tests, other strategies) won't get their randomness perturbed.Reproducibility: same seed → same sequence of "random" picks every time.
  • confidence=1.0/n means "I'm 25% sure" — this is the honest thing to do. It tells the ensemble and logger "I'm literally guessing."When combined in an ensemble, a 25% confidence means this vote barely counts.

Why does this exist? It's the scientific baseline. If your fancy RAG+Agent system scores 26%, you know something is broken. Any real strategy should score significantly above 25%.

Strategy 3

BaselineLLMStrategy — Pure Brain Power

🧠 Core

This is the smart contestant who studied everything in school but brought no notes. They sit down, read the question, and answer purely from memory (their training data). No Google, no calculator, no phone-a-friend. Just raw knowledge baked into their neural network during training.

This strategy sends the question to an LLM (Large Language Model) and gets back an answer. It supports two completely different "decoding paths" depending on how you want the LLM to respond:

How It Looks in the Code

polimibot/strategies/llm_baseline.py
class BaselineLLMStrategy(Strategy):
    """One model, one prompt, one forward pass."""

    def __init__(self, llm, *, style=PromptStyle.ZERO_SHOT,
                 use_score_options=True, ...) -> None:
        if style in _COT_STYLES and use_score_options:
            raise ValueError(
                "CoT requires free generation — score_options reads "
                "the first token, which is reasoning, not the answer."
            )
        self.llm = llm
        self.style = style
        self.use_score_options = use_score_options

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        messages = build_messages(
            inp.question, inp.options,
            category=inp.category, style=self.style,
        )
        if self.use_score_options:
            return self._answer_via_logits(messages)
        return self._answer_via_generation(messages)

    def _answer_via_logits(self, messages) -> StrategyOutput:
        result = self.llm.score_options(messages)
        chosen_index = ord(result.top_letter) - ord("A")
        return StrategyOutput(
            chosen_index=chosen_index,
            confidence=result.top_prob,
            extras={"probs": result.probs, "margin": result.margin},
        )

    def _answer_via_generation(self, messages) -> StrategyOutput:
        response = self.llm.generate(messages, max_new_tokens=...,
                                     temperature=0.0, stop_strings=...)
        idx = parse_answer(response.text)
        parse_ok = idx is not None
        return StrategyOutput(
            chosen_index=idx if parse_ok else 0,
            confidence=0.5 if parse_ok else 0.25,
            is_abstain=not parse_ok,
        )
  • if style in _COT_STYLES and use_score_options: raise — A safety check that prevents a nonsensical combination. If you use Chain-of-Thought, the model's first token is "Let me..." not "B". Reading logits would give gibberish.Fail-fast design: catch misconfigurations at construction time, not per-question.
  • build_messages() formats the question into the correct chat-message shape (system prompt + user prompt) based on the chosen PromptStyle.The prompt module is decoupled from the strategy — you can change prompts without touching strategy code.
  • self.llm.score_options(messages) does a single forward pass and reads the logit (probability) for tokens "A", "B", "C", "D". The highest probability wins.This is the fastest and most informative path — you get actual confidence numbers.
  • parse_answer(response.text) runs a regex cascade to find "Answer: B" or "\boxed{B}" in the generated text. Returns None if it can't find a letter.The model's free text must be machine-parsed — regex is the robust bridge between LLM text and a 0–3 integer.
  • If parsing fails, the strategy sets is_abstain=True instead of guessing. The runner then submits a fallback (default: option A).Honest abstention prevents silent bias — if we defaulted to A, we'd inflate accuracy on questions where A happens to be correct.
Strategy 4

RAGStrategy — The Open-Book Exam

📚 Core

Imagine taking a test, but you're allowed to bring a textbook. You don't read the whole textbook — you quickly flip to the relevant pages (retrieval), skim the key paragraphs (context formatting), and then use your brain to answer (LLM scoring). If your textbook doesn't cover the topic, you quickly Google it on your phone (live search fallback). And next time, you add that new info to your notes (IndexGrower).

RAG (Retrieval-Augmented Generation) is the most powerful single strategy in the system. It combines external knowledge with the LLM's reasoning ability. The pipeline has 6 stages:

The Core Answer Method

polimibot/strategies/rag_strategy.py — answer()
def answer(self, inp: StrategyInput) -> StrategyOutput:
    # 1. Build retrieval query/queries
    category_filter = (
        inp.category.value
        if (self.use_category_filter and inp.category is not None)
        else None
    )

    if self.use_multi_query:
        queries = _build_multi_queries(inp)
        # Retrieve per-query, then RRF-fuse across queries
        ranked_lists = [
            self.retriever.retrieve(q, k=pool_k, **per_query_kwargs)
            for q in queries
        ]
        fused = reciprocal_rank_fusion(ranked_lists, k=pool_k)
        if self.use_reranker:
            passages = self.retriever.rerank_pool(inp.question, fused, k=self.k)
        else:
            passages = fused[:self.k]
    else:
        query = _build_query(inp)
        passages = self.retriever.retrieve(query, **common_kwargs)

    # 2. Score gating — is the retrieval quality good enough?
    top_score = float(passages[0][1]) if passages else 0.0
    gated = (active_threshold is not None
             and (not passages or top_score < active_threshold))

    # 3. Live-search fallback (fires only when offline is gated)
    if gated and self._live_search is not None:
        live_articles = self._live_search.search(inp.question, ...)
        if live_articles:
            # Buffer in IndexGrower for future learning
            if self.index_grower is not None:
                self.index_grower.buffer(article, question_id)
            live_passages = [(chunk, 1.0) for chunk in chunks[:self.k]]

    # 4. Format context (offline passages OR live passages)
    context = _format_context(effective_passages, ...)

    # 5. Build RAG-augmented prompt
    messages = build_messages_with_context(
        inp.question, inp.options, context, ...
    )

    # 6. Score via logits or generate
    if self.use_score_options:
        chosen_index, confidence, ... = self._answer_via_logits(messages)
    else:
        chosen_index, confidence, ... = self._answer_via_generation(messages)
  • In multi-query mode, builds 5 separate queries (question alone + question paired with each option). Each query retrieves its own ranked list, then they're merged with RRF (Reciprocal Rank Fusion).An entity might appear only in one option (e.g., "Who painted the Mona Lisa?" — "Da Vinci" is in option C). Querying with each option separately ensures its supporting article gets found.
  • The category filter restricts retrieval to chunks tagged with the question's category (e.g., only HISTORY chunks for a history question).Prevents noise: a maths question shouldn't pull in random entertainment trivia.
  • Score gating checks if the best passage's score is above a calibrated threshold. If retrieval quality is poor (irrelevant articles), the context is dropped entirely.Bad context is worse than no context — an irrelevant passage can mislead the LLM into a wrong answer.
  • When offline retrieval is gated (scored too low), live Wikipedia search fires as a fallback. The fetched articles are chunked and used as context for this question.The offline index can't cover everything. Live search fills knowledge gaps in real-time.
  • The IndexGrower buffers live-fetched articles. If the game confirms the answer was correct, those articles are permanently added to the index for future questions.Self-improving system: the bot learns from correct answers and gets smarter over time.

Multi-Query Construction

polimibot/strategies/rag_strategy.py — _build_multi_queries()
def _build_multi_queries(inp: StrategyInput) -> list[str]:
    """One query for the question, one per option.
    Returns: [question, "question option_A", "question option_B", ...]
    """
    return [inp.question] + [
        f"{inp.question} {opt}" for opt in inp.options
    ]

Why not just concatenate all options into one query? Because three of the four options are wrong distractors. Averaging them into a single embedding vector means the correct answer's entity gets only ~20% weight — diluted by noise. Separate queries give each option a fair chance, then RRF fusion picks the consensus winners.

Context Formatting (What the LLM Sees)

polimibot/strategies/rag_strategy.py — _format_context()
def _format_context(passages, *, max_passage_chars=800,
                    max_total_chars=2400) -> str:
    """Number passages, add citation headers, truncate at sentence boundaries."""
    parts = []
    used = 0
    for i, (chunk, _score) in enumerate(passages, start=1):
        snippet = _truncate_to_sentence(chunk.text, max_passage_chars)
        header = f"[{i}] {chunk.source} (chunk {chunk.chunk_id})"
        entry = f"{header}\n{snippet}"
        if used + len(entry) > max_total_chars:
            break
        parts.append(entry)
        used += len(entry)
    return "\n\n".join(parts)
  • _truncate_to_sentence cuts passages at a sentence boundary (not mid-word). The LLM never sees dangling fragments that could confuse it.Clean input → better reasoning. A half-sentence like "Caesar crossed the" is useless.
  • Two budgets: max 800 chars per passage, max 2400 chars total. This prevents context from eating the entire LLM input window.LLM context windows are finite. Leave room for the question, instructions, and generation.
  • Each passage gets a numbered header like [1] Julius Caesar (chunk 3, history). The LLM can reference "[Passage 1]" in reasoning.Structured citations help the LLM attribute its reasoning and help humans debug wrong answers.
Strategy 5

ToolStrategy — The Calculator Belt

🔧 Advanced

Imagine a contestant who wears a utility belt. Before using their brain (the LLM), they check their belt: "Do I have a tool for this?" If the question is "What is 15% of 400?", they grab their calculator, compute 60, match it against the options, and answer instantly with 99% confidence. If no tool applies, they put the belt away and think with their brain (fallback LLM). This is the Chain of Responsibility pattern.

ToolStrategy Code

polimibot/strategies/tool_strategy.py
class ToolStrategy(Strategy):
    """Try registered tools in order; fall back to an LLM strategy."""

    def __init__(self, tools: list[Tool], fallback: Strategy) -> None:
        self.tools = tools
        self.fallback = fallback
        tool_names = "+".join(t.name for t in tools)
        self.name = f"tool[{tool_names}|fallback={fallback.name}]"

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        for tool in self.tools:
            if tool.can_handle(inp):
                out = tool.use(inp)
                if out is not None:
                    return out   # tool answered — done
        # All tools abstained
        return self.fallback.answer(inp)
  • for tool in self.tools — tools are tried in priority order. First one that can handle AND produces an answer wins. This is the Chain of Responsibility pattern.Separates "what to try" (tool list) from "how to answer if tools fail" (fallback). Add new tools without touching existing code.
  • tool.can_handle(inp) is a fast guard (no computation). MathsTool checks inp.category == Category.MATHS — instant boolean, no heavy processing.Don't waste time running a calculator on a history question.
  • if out is not None — a tool can return None to say "I tried but couldn't solve this." The chain continues to the next tool.Precision over recall: better to abstain than give a wrong deterministic answer.

MathsTool — The Deterministic Calculator

polimibot/tools/maths_tool.py
class MathsTool(Tool):
    """Deterministic arithmetic solver. Only answers when it can prove the answer."""
    name = "maths_tool"

    def can_handle(self, inp: StrategyInput) -> bool:
        return inp.category == Category.MATHS

    def use(self, inp: StrategyInput) -> Optional[StrategyOutput]:
        # 1. Normalize: "15% of 200" → "15/100*200"
        expr = _extract_expression(inp.question)
        if expr is None:
            return None  # can't parse → abstain

        # 2. Evaluate safely (AST sandbox, no eval())
        try:
            result = safe_eval(expr)
        except Exception:
            return None  # math error → abstain

        # 3. Match against numeric options
        idx = _match_options(float(result), inp.options)
        if idx is None:
            return None  # doesn't match any option → abstain

        return StrategyOutput(
            chosen_index=idx,
            confidence=0.99,   # deterministic = near-certain
            rationale=f"MathsTool: {expr} = {result}",
        )
  • Text normalization converts natural language math ("15% of 200", "square root of 144") into Python arithmetic ("15/100*200", "sqrt(144)") using regex patterns.LLMs sometimes struggle with arithmetic. Deterministic computation is always correct.
  • safe_eval() uses Python's AST (Abstract Syntax Tree) parser to evaluate only whitelisted operations. It's NOT eval() — that would be a security hole.The model's output goes through here. If we used eval(), a malicious expression could execute arbitrary code.
  • confidence=0.99 — a correct computation is near-certain. In an ensemble, this vote will dominate because 0.99 >> 0.50 (LLM confidence).Deterministic answers should outweigh probabilistic ones when both are present.
Strategy 6

AgentStrategy — The Thinking Loop

🤖 Advanced

Think of a contestant who can talk to themselves and use a calculator mid-thought. They read the question, think aloud ("I need to calculate..."), reach for their calculator, get a result, think some more, and finally say their answer. This is the ReAct pattern (Reasoning + Acting): the LLM reasons, calls tools, observes results, and iterates until it has an answer — all within a time budget.

The ReAct Loop

polimibot/strategies/agent_strategy.py
class AgentStrategy(Strategy):
    """ReAct agent: LLM emits CALL: markers; tool results injected as user turns."""

    def __init__(self, llm, *, max_iterations=3, per_turn_tokens=200):
        self.llm = llm
        self.max_iterations = max_iterations

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        deadline = time.monotonic() + inp.time_budget_seconds - 1.0
        messages = self._initial_messages(inp)
        trace = []

        for iteration in range(self.max_iterations + 1):
            if time.monotonic() >= deadline:
                break

            resp = self.llm.generate(messages, max_new_tokens=200, temperature=0.0)
            text = resp.text.strip()
            messages.append({"role": "assistant", "content": text})

            # Branch 1: tool call detected
            call = _extract_call(text)
            if call and time.monotonic() < deadline:
                tool_name, expression = call
                observation = _run_calc(expression)
                messages.append({
                    "role": "user",
                    "content": f"Tool result: {observation}\nNow give your final answer."
                })
                continue   # next iteration

            # Branch 2: final answer detected
            idx = parse_answer(text)
            if idx is not None:
                return StrategyOutput(chosen_index=idx, confidence=0.70)

            # Branch 3: neither — nudge the model
            messages.append({
                "role": "user",
                "content": "Please write your final answer: 'Answer: '."
            })

        # Exhausted — salvage from trace
        return StrategyOutput(chosen_index=0, is_abstain=True)
  • deadline = time.monotonic() + budget - 1.0 — leaves a 1-second safety margin before the hard server deadline. If the loop runs too long, it breaks out and salvages what it can.The game server has a 30-second timer. Exceeding it means automatic wrong answer.
  • _extract_call(text) uses regex + balanced-paren scanning to find CALL: calc(expr) in the LLM's output. The expression can contain nested parentheses like factorial(5).The LLM "decides" to use a tool by emitting a special text pattern — no function-calling API needed.
  • Tool results are injected as a new user message. The LLM sees the conversation grow: its reasoning → tool result → and can now conclude.Multi-turn conversation simulates a "think → act → observe → conclude" loop.
  • If the LLM produces text that's neither a tool call nor a final answer, we nudge it: "Please write your final answer." This prevents infinite spinning.LLMs sometimes ramble. The nudge gives them one more chance to commit to an answer.
  • If all iterations are exhausted, the strategy scans the trace backwards looking for any parseable answer. If none found, it abstains.Graceful degradation: even a timeout shouldn't crash. Try to salvage, then honestly report failure.

Tool Call Detection (Balanced Parens)

polimibot/strategies/agent_strategy.py — _extract_call()
def _extract_call(text: str) -> Optional[tuple[str, str]]:
    """Find 'CALL: tool_name(expr)' handling nested parens correctly."""
    m = re.search(r"CALL\s*:\s*(\w+)\s*\(", text, re.IGNORECASE)
    if not m:
        return None
    tool_name = m.group(1).lower()
    start = m.end()      # first char INSIDE the outer '('
    depth = 1
    for i, ch in enumerate(text[start:]):
        if ch == "(":  depth += 1
        elif ch == ")":
            depth -= 1
            if depth == 0:
                return tool_name, text[start : start + i]
    return None   # unbalanced parens

Why balanced-paren scanning instead of a simple regex? Because expressions can be nested: calc(factorial(5) + sqrt(16)). A regex like \((.+?)\) would stop at the first ) and capture only factorial(5 — broken! The manual scan counts depth to find the matching outer parenthesis.

Strategy 7

EnsembleStrategy — Wisdom of Crowds

🗳️ Orchestration

Imagine asking three experts the same question: a historian, a Wikipedia researcher, and a calculator wizard. Instead of trusting just one, you combine their votes. If two say "B" and one says "C", you go with "B". But it's even smarter — if the calculator is 99% sure and the others are only 50% sure, the calculator's vote counts more. This is ensemble fusion — combining multiple imperfect judges to get a better answer than any single one.

polimibot/strategies/ensemble_strategy.py
class EnsembleStrategy(Strategy):
    """Run sub-strategies sequentially, combine their verdicts."""

    def __init__(self, strategies: list[Strategy], *,
                 weights=None, mode="weighted"):
        self.strategies = strategies
        self.weights = weights or [1.0] * len(strategies)
        self.mode = mode

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        outputs = [s.answer(inp) for s in self.strategies]

        # Drop strategies that abstained
        active = [
            (out, w) for out, w in zip(outputs, self.weights)
            if not out.is_abstain
        ]
        if not active:
            return outputs[0]  # all abstained — nothing better to do

        if self.mode == "weighted":
            return self._weighted_fusion(active)
        return self._vote_fusion(active)

    def _weighted_fusion(self, active):
        combined = {"A": 0.0, "B": 0.0, "C": 0.0, "D": 0.0}
        total_w = sum(w for _, w in active)

        for out, w in active:
            probs = _extract_probs(out)       # get full distribution
            norm_w = w / total_w              # normalize weights to sum=1
            for letter in "ABCD":
                combined[letter] += norm_w * probs[letter]

        best_letter = max(combined, key=lambda l: combined[l])
        return StrategyOutput(
            chosen_index=ord(best_letter) - ord("A"),
            confidence=combined[best_letter],
        )
  • [s.answer(inp) for s in self.strategies] — all strategies are run sequentially (they share one GPU). Parallelism isn't possible here since they compete for the same CUDA resources.Sequential is intentional, not a limitation — GPU can only run one model at a time.
  • Strategies that abstained (e.g., MathsTool on a non-math question) are dropped from the fusion. Their weight is automatically redistributed via normalization.An abstaining strategy saying "I don't know" shouldn't dilute the signal from strategies that DO know.
  • _extract_probs(out) tries to get the full probability distribution from extras["probs"]. If unavailable, it creates a "soft one-hot": chosen letter gets the confidence, rest share the remainder.Graceful fallback: even strategies without full probs can participate meaningfully in fusion.
  • norm_w = w / total_w — weights always sum to 1 after normalization. You can assign weights like [2.0, 1.0, 1.0] to trust the first strategy more, or [1,1,1] for equal trust.Flexible trust allocation without requiring exact values. Just express relative preference.

Why is ensemble better than any single strategy? Different strategies have different blind spots. The LLM might hallucinate on arithmetic; the MathsTool can't handle word problems; RAG fails on rare topics not in the index. By fusing their opinions, you get the union of their strengths — each covers the others' weaknesses.

Strategy 8

TieredStrategy — The Smart Dispatcher

🎯 Orchestration

Think of a hospital triage system. A small cut? → Nurse handles it (cheap, fast). Broken bone? → Doctor (more expensive, more capable). Heart attack? → Specialist surgeon (most expensive, most capable). You don't send every patient to the surgeon — that wastes resources. Similarly, easy quiz questions don't need the full RAG+Ensemble pipeline. A bare LLM can answer "What color is the sky?" in milliseconds. Save the heavy artillery for level-15 questions.

@dataclass(frozen=True)
class TierBreakpoints:
    easy_max_level: int = 5     # levels 1..5 → easy
    medium_max_level: int = 10  # levels 6..10 → medium
                                 # levels 11..15 → hard


class TieredStrategy(Strategy):
    """Route questions to the cheapest strategy that can handle them."""

    def __init__(self, easy, medium, hard, *,
                 breakpoints=TierBreakpoints(),
                 maths_override=None,
                 escalation_threshold=None):
        self.easy = easy
        self.medium = medium
        self.hard = hard
        self.maths_override = maths_override
        self.escalation_threshold = escalation_threshold

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        primary, next_tier = self._route(inp)
        out = primary.answer(inp)

        # Confidence-based escalation
        if (next_tier is not None
            and self.escalation_threshold is not None
            and not out.is_abstain):
            margin = out.extras.get("margin")
            if isinstance(margin, float) and margin < self.escalation_threshold:
                escalated = next_tier.answer(inp)
                return escalated  # use the more expensive answer
        return out

    def _route(self, inp) -> tuple[Strategy, Optional[Strategy]]:
        # Maths override: ALL maths questions → special handler
        if inp.category == Category.MATHS and self.maths_override:
            return self.maths_override, None

        bp = self.breakpoints
        if inp.level <= bp.easy_max_level:
            return self.easy, self.medium      # can escalate to medium
        if inp.level <= bp.medium_max_level:
            return self.medium, self.hard      # can escalate to hard
        return self.hard, None                 # already at max tier
  • _route() maps the question to a (primary, next_tier) pair. The primary runs first. If it's uncertain, the next tier gets a chance.Cost optimization: only spend compute where it's needed. Easy questions get fast answers.
  • Maths override is checked FIRST, regardless of level. A level-1 arithmetic question is better handled by the MathsTool than any LLM.Category-specific routing: some question types have specialized solvers that beat general approaches at all difficulty levels.
  • If margin < escalation_threshold, the cheap strategy wasn't confident enough. The system "escalates" to the next tier (more expensive but more capable).Adaptive compute: spend more time/resources only when the cheap path is uncertain. Confident cheap answers skip the expensive path entirely.
  • At the hard tier, next_tier=None — there's nothing to escalate to. This is the best we've got.Design constraint: escalation stops at the top. The hard strategy is the "last resort" — usually a full ensemble.

The beauty of tiering: In a 15-question game, questions 1–5 are trivial ("What color is the sky?"). Using a full RAG+Ensemble on these wastes 5× the compute and adds latency for no accuracy gain. Tiering gives you the same accuracy as running the hard strategy on everything, but at a fraction of the cost and time.

The Runner: Game Loop

polimibot/runner.py — play_game() simplified
def play_game(client, competition_id, strategy, *, logger=None, 
              fallback_index=0, verbose=True) -> GameResult:
    """Play one full game with the given strategy."""
    
    game = GameAdapter(client, competition_id)
    pacer = _Pacer(min_delay=1.5)  # don't spam the server
    
    while not game.is_over:
        q = game.current_question
        
        # Build input for the strategy
        inp = StrategyInput(
            question=q.text, options=q.options,
            level=q.level, category=category,
            time_budget_seconds=budget,
        )
        
        # Run strategy under watchdog (daemon thread + timeout)
        try:
            out = _call_with_deadline(strategy, inp, budget)
        except _DeadlineExceeded:
            out = None   # timed out → use fallback
        
        chosen_idx = out.chosen_index if out and not out.is_abstain else fallback_index
        
        # Submit answer (rate-limited)
        pacer.wait()
        outcome = game.submit_answer(chosen_idx)
        
        # Learning: confirm or discard live-search articles
        grower = getattr(strategy, "index_grower", None)
        if grower:
            if outcome.correct:
                grower.confirm(f"lvl_{q.level}")   # add to permanent index
            else:
                grower.discard(f"lvl_{q.level}")   # throw away bad info
        
        logger.log_question(...)  # persist everything for analysis
    
    # End of game: flush any learned articles to disk
    if grower:
        grower.flush()
    
    return GameResult(...)
  • _call_with_deadline runs the strategy in a daemon thread. If it exceeds the time budget, the thread is abandoned and the runner submits the fallback answer (default: A).The game server has a hard timer. A timed-out answer is wrong — better to submit a guess than nothing.
  • The _Pacer ensures at least 1.5 seconds between API calls to the server.Politeness: the proof-of-concept server isn't designed for rapid-fire requests. Rate limiting prevents accidental DoS.
  • After each answer, the runner tells the IndexGrower whether it was correct or not. Correct → live-fetched articles join the permanent index. Wrong → they're discarded.Only learn from correct answers. Adding wrong-answer context would poison future retrieval.
  • At game end, grower.flush() persists any learned articles to disk so they survive across sessions.The bot remembers what it learned. Next game starts with a bigger, better knowledge base.

Strategy Composition: How They Nest

The most powerful aspect of this architecture is composability. Strategies can contain other strategies. Here's a typical production configuration:

Typical production wiring (conceptual)
# Layer 1: Atomic strategies
baseline = BaselineLLMStrategy(llm, style=PromptStyle.ZERO_SHOT)
rag      = RAGStrategy(llm, retriever, k=3, use_multi_query=True,
                       use_hybrid=True, use_live_fallback=True,
                       index_grower=grower)
rag_hard = RAGStrategy(llm, retriever, k=5, use_reranker=True,
                       use_multi_query=True, use_hybrid=True)

# Layer 2: Tool composition
maths_tool = MathsTool()
tool_strat = ToolStrategy(tools=[maths_tool], fallback=rag)

# Layer 3: Ensemble (for hard questions)
ensemble = EnsembleStrategy(
    strategies=[rag_hard, tool_strat],
    weights=[2.0, 1.0],  # trust RAG more than tools
    mode="weighted",
)

# Layer 4: Tiered dispatch (the top-level strategy)
final = TieredStrategy(
    easy=baseline,           # levels 1-5: fast LLM
    medium=rag,              # levels 6-10: RAG
    hard=ensemble,           # levels 11-15: full ensemble
    maths_override=tool_strat,  # all maths: calculator first
    escalation_threshold=0.15,  # escalate if margin < 15%
)

# Run the game
result = play_game(client, competition_id=1, strategy=final)
