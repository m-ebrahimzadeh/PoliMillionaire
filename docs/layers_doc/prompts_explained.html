Section 01

Why Prompt Design Matters

In PoliMillionaire, the same LLM answers trivia questions across four categories. The difference between 60% and 85% accuracy often isn't a better model — it's a better prompt.

🎯
Core Philosophy A PromptStyle is an experiment condition, not a magic string. Only the templates.py module knows what text goes to the model. Everything else speaks in terms of the enum — making ablation studies trivial.

The prompt system in polimibot/prompts/templates.py is designed with three principles:

🧪

Ablatable

Swap one enum value → completely different prompting strategy. Same interface, different experiment condition. No code changes needed.

🧩

Composable

System prompt + output instruction + optional few-shot + optional RAG context. Each piece is independent and can be toggled.

✅

Testable

Pure functions that return message lists. No model dependency. Unit tests verify structure without loading a 7B model.

Section 02

The PromptStyle Enum — 5 Experiment Conditions

Each PromptStyle produces a completely different prompt structure. Changing the enum is like flipping a switch between different strategies — the calling code doesn't change at all.

polimibot/prompts/templates.py — the enum 
class PromptStyle(str, Enum):
    """Ablatable prompting variants. Add new ones here; nothing else changes."""
    ZERO_SHOT     = "zero_shot"       # no examples, no CoT — direct "Answer: X"
    ZERO_SHOT_COT = "zero_shot_cot"   # no examples, numbered-step reasoning + \boxed
    FEW_SHOT      = "few_shot"        # 1 curated example per category
    FEW_SHOT_COT  = "few_shot_cot"    # 1 example with reasoning trace + \boxed
    ELIMINATION   = "elimination"     # option-by-option scaffolding + \boxed

Zero-Shot zero_shot

No examples, no reasoning. The model must answer directly in one letter.

Answer: B
✓ logit scoring

Zero-Shot CoT zero_shot_cot

Step-by-step reasoning chain, then commit to an answer.

Step 1: The question asks... Step 2: Recall that... Step 3: Therefore... \boxed{B}
✓ generation only

Few-Shot few_shot

One curated example per category, then the real question. Direct answer.

Answer: B
✓ logit scoring 1 example

Few-Shot CoT few_shot_cot

One example with full reasoning trace, then the real question with CoT.

Steven Spielberg directed Jaws (1975) and Schindler's List (1993). \boxed{B}
✓ generation only 1 example

Elimination elimination

Evaluate each option systematically, eliminating wrong answers before committing. Forces the model to consider every option rather than anchoring on the first plausible one.

A: Martin Scorsese — directed Goodfellas, not Jaws. Wrong. B: Steven Spielberg — directed both Jaws and Schindler's List. Correct. C: Francis Ford Coppola — directed The Godfather, not these. Wrong. D: Stanley Kubrick — directed 2001, not these. Wrong. \boxed{B}
✓ generation only
⚡
Key Rule: Logit Scoring vs Generation Only ZERO_SHOT and FEW_SHOT support logit scoring (use_score_options=True). The CoT and Elimination styles must use free generation because the first predicted token is the start of the reasoning trace, not the answer letter. The code enforces this with a hard error.
Section 03

System Prompts — Category-Specific Instructions

Each category gets a different system prompt that teaches the model the gotchas of that domain. These are instructive, not flattering — generic praise ("You are an expert") leaves accuracy on the table.

💡
Design Insight The system prompts warn about specific distractor patterns the trivia game uses. For History: "similar-sounding dynasties, namesake successors." For Science: "units confusion, necessary vs sufficient conditions." This primes the model to double-check exactly the things the distractors exploit.
system Category: ENTERTAINMENT
You are answering multiple-choice trivia on film, music, television, and pop culture. Trivia distractors often pair real co-stars from the same era, films by the same director, or bands sharing members. Verify the specific year, director, album, or actor before committing.
system Category: HISTORY
You are answering multiple-choice trivia on ancient history, classical civilisations, and political history. Watch for plausible distractors that confuse adjacent centuries, similar-sounding dynasties, or namesake successors (e.g. Caesar vs Augustus, Henry V vs Henry VIII). Verify the specific century or reign before committing.
system Category: SCIENCE
You are answering multiple-choice trivia on biology, chemistry, physics, and the natural world. Be careful with units (atoms vs molecules, kg vs g, joules vs calories) and distinguish necessary from sufficient conditions in causal claims.
system Category: MATHS
You are a careful mathematician. Compute precisely — do not guess. Verify each calculation before committing.
system Category: None (fallback)
You are a careful, knowledgeable quiz contestant. From four options labelled A, B, C and D, pick the single best answer.

Section 04

Output Instructions — Direct, CoT, Elimination

The output instruction tells the model how to format its answer. Different styles require different formats — and the format must be parseable back to a single letter.

The Three Instruction Blocks

_DIRECT Used by: ZERO_SHOT, FEW_SHOT
Begin your reply with 'Answer:' followed by exactly one letter from A, B, C, D. Do not include any text before 'Answer:'.
_COT Used by: ZERO_SHOT_COT, FEW_SHOT_COT
Solve step by step using this structure: Step 1: Restate what the question asks. Step 2: Compute, eliminate, or recall the relevant fact. Step 3: End with \boxed{X} on its own line, where X is one of A, B, C, D.
_ELIMINATION Used by: ELIMINATION
Evaluate each option in one short sentence, then commit: A: <why right or wrong> B: <why right or wrong> C: <why right or wrong> D: <why right or wrong> Then on a new line write \boxed{X} with the best answer.
polimibot/prompts/templates.py — instruction selector 
def _instruction_for(style: PromptStyle) -> str:
    """Pick the output-instruction block for the given style."""
    if style == PromptStyle.ELIMINATION:
        return _ELIMINATION
    if style in (PromptStyle.ZERO_SHOT_COT, PromptStyle.FEW_SHOT_COT):
        return _COT
    return _DIRECT
📐
Why \boxed{X}? \boxed{X} is the native answer format for Qwen-Math and DeepSeek-Math RL models. Using it as the termination signal means: (1) the model produces it naturally without extra training, (2) it serves as an early-stop signal (cuts off "Hope this helps!" chatter), and (3) it's unambiguous for the regex parser.
Section 05

Few-Shot Examples — Teaching by Demonstration

For FEW_SHOT and FEW_SHOT_COT styles, one hand-curated example per category is injected before the real question. These examples are carefully designed to teach reasoning patterns, not just answer format.

Design Principles for Examples

  • Exercise the right reasoning — modular arithmetic for Maths, date disambiguation for History
  • Avoid word-overlap — if the question mentions the correct option's noun, the model learns a brittle "pick the repeated word" heuristic
  • Match the output format — CoT examples include rationale + \boxed{X}; direct examples just say Answer: X

How Examples Become Messages

Each example is turned into a user + assistant message pair. The assistant's response format exactly matches the output instruction the model just received — so the example teaches the format, it doesn't contradict it:

polimibot/prompts/templates.py — example formatting 
def _example_answer_format(style: PromptStyle, rationale: Optional[str], letter: str) -> str:
    """How a few-shot assistant turn should phrase its answer."""
    if style in (PromptStyle.ZERO_SHOT_COT, PromptStyle.FEW_SHOT_COT) and rationale:
        return f"{rationale}\n\\boxed{{{letter}}}"
    return f"Answer: {letter}"

# For FEW_SHOT (direct): assistant says "Answer: B"
# For FEW_SHOT_COT: assistant says "Steven Spielberg directed Jaws...\n\boxed{B}"

polimibot/prompts/templates.py — build_messages() 
def build_messages(
    question: str,
    options: Sequence[str],
    *,
    category: Optional[Category] = None,
    style: PromptStyle = PromptStyle.ZERO_SHOT,
) -> List[Dict[str, str]]:
    """Assemble chat messages for a single MCQ question.

    This is the only function the strategy layer calls.
    Swap the style → different experiment condition, same interface.
    """
    if len(options) != 4:
        raise ValueError(f"Expected 4 options, got {len(options)}")

    messages: List[Dict[str, str]] = [
        {"role": "system", "content": _system_prompt(category)}
    ]

    # Inject the curated example if this is a few-shot style
    if style in (PromptStyle.FEW_SHOT, PromptStyle.FEW_SHOT_COT):
        example = _FEW_SHOT_BANK.get(category) if category else None
        if example:
            messages.extend(_few_shot_turns(example, style=style))

    # The actual question
    messages.append({"role": "user", "content": _user_turn(question, options, style=style)})
    return messages

Section 07

RAG Context Injection — build_messages_with_context()

When the RAG strategy retrieves relevant passages, they need to be injected into the prompt. build_messages_with_context() handles this with a careful ordering rationale.

🧠
Ordering Rationale Chat-tuned models attend most strongly to the most recent tokens. The question + options come BEFORE the context, so the question never has to compete with retrieval for attention. The context is framed as "reference material (may or may not be relevant)" — hedging avoids pulling the model toward fabricated answers from off-topic retrievals.
polimibot/prompts/templates.py — build_messages_with_context() 
def build_messages_with_context(
    question: str,
    options: Sequence[str],
    context: str,            # pre-formatted retrieval results
    *,
    category: Optional[Category] = None,
    style: PromptStyle = PromptStyle.ZERO_SHOT,
) -> List[Dict[str, str]]:
    """Like build_messages, but appends retrieved passages after the question."""
    if not context:
        return build_messages(question, options, category=category, style=style)

    instr = _instruction_for(style)

    user_content = (
        f"Question: {question}\n\n"
        f"Options:\n{_format_options(options)}\n\n"
        f"Reference material (may or may not be relevant):\n{context}\n\n"
        f"{instr}"
    )

    messages = [
        {"role": "system", "content": _system_prompt(category)},
        {"role": "user",   "content": user_content},
    ]
    return messages

Key Design Choice: No Few-Shot with RAG Notice that build_messages_with_context() does NOT inject few-shot examples, even for FEW_SHOT or FEW_SHOT_COT styles. The context already uses most of the attention budget — adding examples on top would push the model's attention away from the retrieved passages that actually contain the answer.
Section 08

Answer Parsing — The Regex Cascade

When the LLM generates free text (CoT/Elimination modes), we need to extract the final answer letter. The parse_answer() function uses a carefully ordered cascade of regex patterns.

The Problem

LLMs are creative with their output format. The same answer might appear as:

  • \boxed{B} — Qwen/DeepSeek Math format
  • Answer: B — direct format
  • The final answer is B — verbose phrasing
  • Option B — referring to the option
  • B. at the start of a line
  • Just B somewhere in the text (last resort)

The Cascade (Most Specific → Least Specific)

1

\boxed{X} — highest priority

Regex: \\boxed\{\s*([A-D])\s*\} — Catches the Math-RL native format. Placed first so it wins against any earlier letters in the reasoning trace.

2

Answer: X or answer - X

Regex: answer\s*[:\-]?\s*\(?([A-D])\)? — The standard direct-format answer.

3

final answer is X / correct answer X

Regex: \b(?:final|correct)\s+answer\s*(?:is)?\s*[:\-]?\s*\(?([A-D])\)? — Verbose phrasing patterns.

4

option X

Regex: \boption\s+\(?([A-D])\)? — "I'll go with option B" style.

5

X. or X) at line start

Regex: ^([A-D])\s*[\.\)] (multiline) — "B." or "B)" at the beginning of a line.

6

Bare letter (LAST match wins)

Regex: \b([A-D])\b — Critical: takes the last match, not the first! So "Let A be the area... the answer is C" correctly picks C, not A.

polimibot/prompts/templates.py — parse_answer() 
def parse_answer(text: str) -> Optional[int]:
    """Extract 0-based option index from model output. Returns None if unparseable.

    Cascade:
      1. Structured patterns (\\boxed{X}, "Answer: X", "final answer is X",
         "Option X", line-leading "X." / "X)") — first match wins.
      2. Bare letter \\b([A-D])\\b — LAST match wins (so a CoT setup like
         "Let A be the area, so the answer is C" picks C, not A).

    Converts letter → 0-based index: A→0, B→1, C→2, D→3.
    """
    if not text:
        return None
    for pattern in _STRUCTURED_PATTERNS:
        m = pattern.search(text)
        if m:
            return ord(m.group(1).upper()) - ord("A")
    # Bare letter — last match wins
    matches = _BARE_LETTER.findall(text)
    if matches:
        return ord(matches[-1].upper()) - ord("A")
    return None   # unparseable → strategy abstains
⚠️
On Parse Failure When parse_answer() returns None, the strategy marks the response as is_abstain=True. The runner then substitutes its fallback index (default: 0). The ensemble system redistributes weight to surviving strategies. This is better than silently defaulting to "A" which would bias accuracy.
Section 09

LLM Integration — Logit Scoring vs Free Generation

The prompt messages produced by build_messages() are consumed by two fundamentally different LLM methods. Understanding the difference is key to understanding why certain styles only work with certain methods.

Method A: score_options() logit scoring

A single forward pass that measures the probability the model assigns to each answer letter at the next-token position:

polimibot/models/llm.py — score_options() core logic 
def score_options(self, messages, letters=("A", "B", "C", "D")):
    """One forward pass → probabilities over answer letters.

    The prompt ends with "Answer: " — the next token IS the answer letter.
    We probe logits[0, -1, :] for the token IDs of A, B, C, D.
    """
    # Apply chat template: adds special tokens like <|im_start|>assistant\n
    prompt = self._apply_template(messages, add_generation_prompt=True)
    prompt = prompt + "Answer: "   # ← the model completes THIS

    inputs = self._tokenizer(prompt, return_tensors="pt").to(self._model.device)

    with torch.inference_mode():
        last_logits = self._model(**inputs).logits[0, -1, :]

    # Find token IDs for " A", " B", " C", " D" (leading space variant)
    letter_ids = {}
    for l in letters:
        ids_with_space = self._tokenizer.encode(" " + l, add_special_tokens=False)
        ids_bare = self._tokenizer.encode(l, add_special_tokens=False)
        letter_ids[l] = ids_with_space[0] if len(ids_with_space) == 1 else ids_bare[0]

    # Softmax over just these 4 logits
    raw = torch.tensor([last_logits[tid] for tid in letter_ids.values()])
    probs = torch.softmax(raw, dim=0)
    # → {"A": 0.05, "B": 0.89, "C": 0.03, "D": 0.03}
⚡
Why Leading-Space Variant? After "Answer: ", the model predicts " B" (with a leading space) not "B". Most tokenizers encode these differently. The code tries the space-prefixed variant first, falls back to bare letter. Getting this wrong means probing the wrong token ID → garbage probabilities.

Method B: generate() free generation

polimibot/models/llm.py — generate() with stop strings 
def generate(self, messages, *, max_new_tokens=None, temperature=0.0, stop_strings=None):
    """Generate free text from a chat-formatted message list.

    stop_strings: when produced, terminate generation.
    E.g. [r"\\boxed{A}", r"\\boxed{B}", ...] cuts off post-answer chatter.
    """
    prompt = self._apply_template(messages, add_generation_prompt=True)
    inputs = self._tokenizer(prompt, return_tensors="pt").to(self._model.device)

    gen_kwargs = dict(
        max_new_tokens=max_new_tokens or self.spec.max_new_tokens,
        do_sample=(temperature > 0),
        temperature=temperature if temperature > 0 else None,
    )
    if stop_strings:
        gen_kwargs["stop_strings"] = list(stop_strings)
        gen_kwargs["tokenizer"] = self._tokenizer

    with torch.inference_mode():
        out = self._model.generate(**inputs, **gen_kwargs)

    new_tokens = out[0][n_in:]
    text = self._tokenizer.decode(new_tokens, skip_special_tokens=True)
    return LLMResponse(text=text, ...)

class BaselineLLMStrategy(Strategy):
    """One model, one prompt, one forward pass."""

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        # Build the prompt messages using the configured style
        messages = build_messages(
            inp.question,
            inp.options,
            category=inp.category,
            style=self.style,
        )
        # Route to the appropriate decoding method
        if self.use_score_options:
            return self._answer_via_logits(messages)
        return self._answer_via_generation(messages)
