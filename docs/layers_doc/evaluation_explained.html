Section 1

Why Evaluate Offline?

Imagine you're training for a quiz show. You wouldn't change your study method after each live game — the sample size is too small and the questions are random. Instead, you'd collect past questions with known answers (a "practice test"), then test every new study method on the same practice test. That way, differences in scores reflect your method, not luck of the draw. This is exactly what offline evaluation does.

The PoliMillionaire game server reveals whether an answer is correct only after you submit it. You can't re-ask the same question. This creates three problems:

ProblemWhy it's badSolution
Can't replay questionsCan't compare strategies on the same questionsMine past games → frozen gold set
Small sample per game15 questions per game is too noisy for statisticsAggregate hundreds of gold items
Network dependencyServer downtime blocks developmentEvaluate 100% offline against gold set

The cardinal rule: Build your gold set ONCE from baseline runs, then FREEZE it. Never re-harvest mid-experiment — that would silently grow your test set and invalidate all previous comparisons.

Section 2

The Gold Set — Your Frozen Test Bank

Think of the gold set as a medical study's patient records. You play games (treat patients), record outcomes (which answers were correct), and after enough data, you freeze that dataset. Every future treatment (strategy) is tested on the same patient cohort — so any improvement is real, not sampling noise.

def harvest_gold_set(runs_dir: Path) -> list[GoldItem]:
    """Mine all *.jsonl run files → GoldItems."""
    confirmed = {}   # key → GoldItem (direct confirmation)
    wrong_indices = {}  # key → set of wrong indices

    for fp in sorted(runs_dir.glob("*.jsonl")):
        for rec in load_jsonl(fp):
            key = f"{cid}|{text}"
            if is_correct is True and key not in confirmed:
                confirmed[key] = GoldItem(...)   # Direct hit!
            elif is_correct is False:
                wrong_indices[key].add(idx)      # Elimination evidence

    # Elimination pass: 3 wrong → 4th is gold
    for key, wrong in wrong_indices.items():
        if key not in confirmed and len(wrong) == 3:
            remaining = set(range(4)) - wrong    # only one left!
            confirmed[key] = GoldItem(correct_index=remaining.pop())

    return list(confirmed.values())
  • If we submitted answer B and the server said "correct" → we know B is the right answer. One confirmation is enough.Most gold items come from direct correct answers during baseline runs.
  • If we've seen A wrong, B wrong, and C wrong across different games → D must be correct (process of elimination). We never need to have gotten it right ourselves!Recovers gold labels from repeated failures — every wrong answer is useful data.
  • Same question can appear in multiple games. We deduplicate by (competition_id, question_text) — first confirmation wins.No double-counting. Each question appears exactly once in the gold set.

Section 3

The Wrong Set — Your Error Archive

If the gold set is your honor roll, the wrong set is your error log. Every time the bot gets a question wrong, that question becomes a data point about its weaknesses. A wrong-set item says: "The bot chose option B on this question, and it was wrong." With enough of these, you can map failure modes by category, level, and competition — and target improvements precisely where they matter.

The wrong set is symmetric to the gold set: both are mined from the same run logs, both are deduplicated, both store options in their original server order with index-based labels. The difference is what they capture:

How Wrong Items Are Mined — Two-Pass Harvesting

Wrong items are harvested with a 2-pass algorithm that first collects all confirmed wrong answers, then tries to recover the correct answer through cross-referencing:

polimibot/eval/wrong_set.py — harvest_wrong_set()
def harvest_wrong_set(runs_dir: Path) -> list[WrongItem]:
    """Mine all *.jsonl run files → WrongItems.

    Two-pass algorithm:
      Pass 1 — collect every confirmed wrong answer (correct=False).
      Pass 2 — cross-reference to recover correct_index where possible.
    """
    # Pass 1: collect wrong answers
    meta    = {}   # key → first-seen metadata (options, competition_id, level, …)
    wrong   = {}   # key → set of wrong indices seen for this question
    correct = {}   # key → correct_index, if we ever saw a correct answer

    for fp in sorted(runs_dir.glob("*.jsonl")):
        for rec in load_jsonl(fp):
            if rec.get("run_kind") != "question":
                continue
            key = f"{cid}|{text}"

            # Cache metadata from the first sighting of this question
            if key not in meta:
                meta[key] = {"options": tuple(rec.get("options", [])), ...}

            if is_correct is False:
                wrong.setdefault(key, set()).add(chosen_index)   # wrong answer
            elif is_correct is True:
                if key not in correct:
                    correct[key] = chosen_index                  # known correct

    # Pass 2: build WrongItems, filling correct_index where possible
    items = []
    for key, wrong_idxs in wrong.items():
        m = meta[key]
        # Method A: direct cross-reference
        recovered = correct.get(key, -1)
        # Method B: elimination (3 of 4 seen wrong → 4th must be correct)
        if recovered == -1 and len(wrong_idxs) == 3:
            remaining = set(range(4)) - wrong_idxs
            recovered = remaining.pop()

        for idx in wrong_idxs:
            items.append(WrongItem(
                options=m["options"],
                wrong_index=idx,
                correct_index=recovered,
                ...
            ))
    return items
  • Iterates over all run logs once. Every record with correct=False is a confirmed wrong answer — stored by (competition_id, question_text, wrong_index).A single wrong-answer record is all we need. We know the bot got it wrong.
  • If the same question was answered correctly in any run (possibly a different session), we borrow that correct_index for all wrong items about this question.Cross-run recovery: one correct answer anywhere unlocks the correct label for all wrong-answer records of the same question.
  • If 3 of the 4 options have been seen as wrong answers across runs, the remaining option must be correct — process of elimination. No correct-answer record needed.Recovers the correct label from pure failure data. The more the bot fails, the more ground truth it inadvertently reveals.
  • When neither Method A nor B applies, correct_index stays -1 (unknown). These items are still valuable for failure-rate analysis, even without a correct label.Partial information is still information. Filter by correct_index != -1 when you need complete items.

Option Order Guarantee

Option order is NEVER changed anywhere in the pipeline. The server sends options as a JSON array; Option.from_dict() parses them in iteration order; GameAdapter extracts tuple(opt.text for opt in q.options); the runner logs them as list(q.options); the harvesters cache them as tuple(rec.get("options", [])); and the WrongItem/GoldItem frozen dataclasses hold them as an immutable tuple. No sort, shuffle, or permutation is ever applied to the options array. The wrong_index and correct_index integers are always 0-based indices into this same-order tuple (0=A, 1=B, 2=C, 3=D).

LayerFileHow options are transferred
Server → API clientmillionaire_client/models.py[Option.from_dict(opt) for opt in data["options"]] — list comprehension, preserves JSON array order
API client → Game adapterpolimibot/game/adapter.pytuple(opt.text for opt in q.options) — generator, preserves iteration order
Game adapter → Strategypolimibot/runner.pyStrategyInput(options=q.options, …) — direct pass-through
Strategy → Log filepolimibot/runner.pyoptions=list(q.options) — tuple → list, same order
Log file → Wrong/Gold Setwrong_set.py / gold_set.pytuple(rec.get("options", [])) — JSON array → tuple, same order
Section 4

The Evaluator — Replaying Questions Offline

Think of it as a driving simulator. Instead of putting a new driver on real roads (live games with consequences), you replay recorded traffic scenarios in a controlled environment. The driver (strategy) doesn't know it's a simulation — it receives the same inputs and makes the same decisions. You measure their performance without risk.

polimibot/eval/evaluator.py — evaluate_strategy()
def evaluate_strategy(
    strategy: Strategy,
    gold_set: list[GoldItem],
    *,
    verbose: bool = True,
) -> EvalReport:
    """Replay gold_set through strategy offline. No live API calls."""
    samples = []

    for i, item in enumerate(gold_set):
        # Build the same StrategyInput the runner would build
        inp = StrategyInput(
            question=item.question_text,
            options=item.options,
            level=item.level,
            category=item.category,
            competition_id=item.competition_id,
        )
        # Time the strategy
        t0 = time.monotonic()
        out = strategy.answer(inp)
        elapsed = time.monotonic() - t0

        # Compare against ground truth
        correct = (out.chosen_index == item.correct_index)
        samples.append(EvalSample(
            correct=correct,
            confidence=out.confidence,
            elapsed_seconds=elapsed,
            ...
        ))

    return _build_report(strategy.name, samples)
  • The evaluator builds the exact same StrategyInput the runner would build during a live game. The strategy can't tell the difference.Fair comparison: strategies are measured under identical conditions.
  • time.monotonic() captures wall-clock latency per question — used to compute p50/p95 percentiles in the report.Latency matters in production: the server has a 30-second timeout per question.
  • out.chosen_index == item.correct_index — binary correct/wrong. No partial credit.Simple, unambiguous metric. Either you got it right or you didn't.

Calibration — "Am I as Sure as I Think?"

Imagine a weather forecaster. When they say "90% chance of rain," it should actually rain 9 out of 10 times they make that prediction. If it only rains 6 out of 10 times → they're overconfident. If it rains all 10 times → they're underconfident. A perfectly calibrated forecaster's stated probability matches reality. The same applies to our bot's confidence scores.

Expected Calibration Error (ECE)

ECE measures how well the bot's confidence aligns with its actual accuracy:

polimibot/eval/calibration.py
def compute_calibration(confidences, corrects, n_bins=10) -> CalibrationResult:
    """Bin predictions by confidence; compare mean confidence to accuracy per bin."""
    bins = np.linspace(0.0, 1.0, n_bins + 1)   # [0, 0.1, 0.2, ..., 1.0]
    conf_arr = np.asarray(confidences, dtype=float)
    corr_arr = np.asarray(corrects, dtype=float)

    bin_confs, bin_accs, bin_counts = [], [], []
    for lo, hi in zip(bins[:-1], bins[1:]):
        mask = (conf_arr >= lo) & (conf_arr < hi)
        n = mask.sum()
        if n == 0:
            bin_confs.append((lo + hi) / 2)
            bin_accs.append(0.0)
        else:
            bin_confs.append(conf_arr[mask].mean())   # what we SAID
            bin_accs.append(corr_arr[mask].mean())    # what actually HAPPENED

    # ECE = weighted average of |accuracy - confidence| per bin
    ece = sum(cnt * abs(acc - conf)
              for cnt, acc, conf in zip(bin_counts, bin_accs, bin_confs)) / total

    return CalibrationResult(ece=ece, bin_confidences=..., bin_accuracies=...)
  • Predictions are grouped into 10 equal-width confidence buckets: [0-10%), [10-20%), ..., [90-100%].Groups similar-confidence predictions together so we can check if their actual accuracy matches.
  • For each bin: avg_confidence is what the bot CLAIMED, avg_accuracy is what ACTUALLY happened. The gap is the calibration error for that bin.A gap of 0 means perfect calibration. A large positive gap means overconfidence.
  • ECE is the weighted average of gaps across all bins (weighted by bin count). A single overconfident prediction on an easy question doesn't dominate the score.Summarizes calibration quality in a single number: 0 = perfect, 0.5 = terrible.

Section 6

Retrieval Evaluation — Does the Right Article Show Up?

Imagine a librarian helping you study. You ask "Who crossed the Rubicon?" and the librarian brings you 3 books. The retrieval evaluation asks: "Was the Julius Caesar book in those 3?" If it was in position 1 → perfect recall@1. If it was in position 3 → still recall@3 success, but rank matters. If it wasn't in the top 3 at all → retrieval failure, even if the LLM might have known the answer anyway.

def evaluate_retrieval(retriever, items, *, ks=(1, 3, 5, 10),
                       query_fn=_default_query, use_category_filter=True,
                       use_reranker=False, use_hybrid=False) -> RetrievalReport:
    """Compute recall@k and MRR for retriever against labeled items."""
    for item in items:
        if not item.is_labeled:
            continue  # skip unlabeled items

        query = query_fn(item)
        hits = retriever.retrieve(query, k=max(ks), category=...)

        # Unique titles in rank order (dedup multiple chunks from same article)
        unique_titles = [chunk.source for chunk, score in hits if not seen]

        # Did the gold article appear in the ranked list?
        found_at = None
        for rank, title in enumerate(unique_titles, start=1):
            if title == item.gold_article_title:
                found_at = rank
                break

    # Compute metrics
    recall_at = {k: (found_in_top_k / n_labeled) for k in ks}
    mrr = mean(1/rank for each found item)

    return RetrievalReport(recall_at=recall_at, mrr=mrr, by_category=...)
  • Only items where a human has filled in gold_article_title contribute to recall scores. Unlabeled items are counted but skipped.Can't measure recall without ground-truth labels. Partial labeling is fine — the harness handles it gracefully.
  • The query_fn parameter lets you test different query recipes (question-only, question+options, HyDE) on the same labeled set without re-labeling.Ablation-friendly: change one variable (the query), keep everything else fixed.
  • Multiple chunks from the same article count as ONE hit at the rank of the first appearance. This prevents an article with 5 chunks from inflating recall.We care about article-level recall, not chunk-level. Did the RIGHT source show up?
  • Mean Reciprocal Rank: if the gold article is at rank 3, its contribution is 1/3. Lower ranks contribute less. MRR rewards finding the right article EARLY.Being at rank 1 (MRR contribution = 1.0) is much better than rank 10 (MRR contribution = 0.1).

Section 7

Ablation Scripts — Controlled Experiments

Like a scientific experiment. You change ONE variable at a time and measure the effect. Does RAG help? (eval_rag: baseline vs RAG, same LLM). Does the calculator help on maths? (eval_tools: baseline vs tool, same maths questions). Does ensemble fusion help on hard questions? (eval_ensemble: single strategies vs ensemble, same hard subset). Each script is a controlled A/B test.

Anatomy of an Ablation Script

scripts/eval_rag.py — structure
def main():
    # 1. Load gold set (frozen test bank)
    gold = load_gold_set(PATHS.eval_dir / "gold_set.jsonl")

    # 2. Optionally filter by category
    if args.categories:
        gold = [g for g in gold if g.category in keep]

    # 3. Build LLM ONCE (shared by all strategies — controlled variable)
    llm = LLM.load(LLMSpec(model_id=args.model))

    # 4. Build strategies using the SAME LLM
    baseline = BaselineLLMStrategy(llm, style=style)
    rag      = RAGStrategy(llm, retriever, k=args.k, ...)

    # 5. Evaluate each and persist reports
    baseline.warm_up()
    report_baseline = evaluate_strategy(baseline, gold)
    save_report(report_baseline, name="baseline_zs__qwen", eval_dir=PATHS.eval_dir)

    report_rag = evaluate_strategy(rag, gold)
    save_report(report_rag, name="rag__qwen", eval_dir=PATHS.eval_dir)

    # 6. Print side-by-side comparison (delta table)
    _print_comparison(report_baseline, report_rag)
    # Shows: Accuracy  68.5%  →  78.2%  (+9.7% ✓)
    #        ECE       0.12   →  0.08   (-0.04 ✓)
    #        Latency   0.5s   →  1.2s   (+0.7s ✗)
  • ALL strategies share the same loaded LLM instance. This ensures any accuracy difference comes from the strategy logic, not model randomness.Scientific method: control all variables except the one you're testing.
  • Reports are saved to data/eval/ immediately after each evaluation. If the script crashes halfway, you don't lose the first strategy's results.Crash-resilient design: partial results are always available.
  • The comparison table shows direction + magnitude of change with ✓/✗ markers. Accuracy up = good (✓). Latency up = bad (✗). ECE down = good (✓).At-a-glance understanding: did this change help or hurt?

def build_leaderboard(eval_dir: Path) -> pd.DataFrame:
    """Scan eval_dir for *.json strategy reports → sorted DataFrame."""
    rows = []
    for p in sorted(eval_dir.glob("*.json")):
        row = _parse_report(p)   # extract: strategy, accuracy, ece, latency, n
        if row:
            rows.append(row)

    df = pd.DataFrame(rows, columns=[
        "strategy", "accuracy", "ece", "latency_p50_s", "latency_p95_s", "n"
    ])
    # Sort by accuracy descending — best strategy first
    return df.sort_values("accuracy", ascending=False)

Logging — The Source of All Truth

Think of run logs as a ship's black box. Every question asked, every answer given, every timing, every confidence score — all recorded in real-time to a crash-safe file. After the game, you can reconstruct exactly what happened, mine gold labels, compute calibration, and debug wrong answers. The logs are the raw material that powers the entire evaluation ecosystem.

The RunLogger

polimibot/logging_utils.py
class RunLogger:
    """JSONL append-only logger. Use as a context manager."""

    def __init__(self, runs_dir, *, run_id, extra=None):
        # Filename: run_20260514_012345_baseline_zs.jsonl
        ts = datetime.now(UTC).strftime("%Y%m%d_%H%M%S")
        self.path = runs_dir / f"run_{ts}_{safe_id}.jsonl"

    def __enter__(self):
        self._fh = self.path.open("a", encoding="utf-8")
        self._write(RunManifest(run_id=..., git_sha=...))  # first line
        return self

    def __exit__(self, *args):
        self._fh.flush()
        os.fsync(self._fh.fileno())   # crash-safety: force to disk
        self._fh.close()

    def log_question(self, rec: QuestionRecord):
        self._write(rec)

    def log_summary(self, rec: GameSummaryRecord):
        self._write(rec)

    def _write(self, rec):
        self._fh.write(json.dumps(asdict(rec)) + "\n")
        self._fh.flush()  # tail -f friendly
  • First line captures git SHA, Python version, platform, strategy name. Enough to reproduce the exact conditions of this run.Reproducibility: months later, you can recreate the exact environment that produced these results.
  • os.fsync() forces the file to disk on close. Even if the process crashes, all committed data is safe.Crash safety: a mid-game crash doesn't lose the questions already played.
  • Each _write() calls flush() immediately. You can tail -f the file and watch questions arrive in real-time during a live game.Live observability: watch the bot play in real-time from another terminal.
  • Every record has a run_kind field ("manifest", "question", "summary"). Readers filter on this to extract only what they need.Self-describing format: one file, multiple record types, trivial to parse.

Why JSONL (not CSV or SQLite)? JSONL is append-only (no schema migration), human-readable (open in any editor), streamable (process line-by-line without loading all into memory), and crash-safe (each line is independent — corruption of one line doesn't kill the file). For experimental logging, it's the sweet spot.

Chain-of-thought was ruled out by the data. For the same model, zero-shot vs. CoT leaves accuracy unchanged (95.7%) but blows calibration error from 0.038 → 0.46 and latency from 1.5 s → 27 s. Logit-scoring the four option tokens in a single forward pass is both faster and far better calibrated.

Limitations. News is under-evaluated — the Guardian free tier rate-limits requests, so only a handful of dated news questions could be harvested. The gold set is mined from confirmed-correct answers, so absolute accuracies are optimistic and should be read as relative comparisons. Maths tops out near 88% on multi-step word problems, and the Philosophy/News samples are thin.
