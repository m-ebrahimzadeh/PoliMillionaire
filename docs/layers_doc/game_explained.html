Section 01

The Game Show — What Are We Playing?

PoliMillionaire is "Who Wants to Be a Millionaire?" — but the contestant is an AI bot. The server asks questions. The bot has 30 seconds to answer. Correct → advance to next level and earn more money. Wrong → game over (but keep a "safe amount" from milestone levels). Reach level 15 → maximum prize.

The Money Pyramid

Each level corresponds to a prize amount. Getting a question wrong drops you back to the last "milestone" (safe amount). The pyramid structure creates increasing stakes:

Level 1 €100
Level 2 €200
Level 5 ⭐ safe €1,000
Level 6 €2,000
Level 10 ⭐ safe €32,000
Level 11-14 €64K–€500K
Level 15 🏆 €1,000,000

Section 02

Architecture Layers — Why Three Separate Layers?

Imagine a phone interpreter at a diplomatic summit. The diplomat (Strategy) speaks only English and doesn't know foreign protocol. The interpreter (Adapter) translates between English and the foreign language. The phone (API Client) handles the actual transmission. Separating these three concerns means you can change any one without touching the others: swap the phone, change the interpreter, or replace the diplomat independently.

The Adapter Pattern enforces a strict boundary. The rule: "Anywhere outside adapter.py importing millionaire_client is a code smell." This means all 30+ strategies, the evaluator, the logging system — none of them ever touch the API client directly. Benefits: (1) you can test strategies with mock game adapters; (2) if the API changes, you fix only one file; (3) strategies are clean and focused.

Section 03

The HTTP Client — Talking to the Server

Think of it like a bank teller system. You (the application) give orders to the teller (MillionaireClient). The teller handles all the paperwork: authentication tokens, request formatting, error translation, and response parsing. You never deal with raw HTTP — just clean Python method calls.

Authentication: Cookie-Based Sessions

millionaire_client/auth.py + base.py — login flow
class AuthModule:
    def login(self, username: str, password: str) -> User:
        # POST /api/auth/login  → server sets "polimillionaire_auth" cookie
        response = self._client.post(
            "/api/auth/login",
            data={"username": username, "password": password},
            auth_required=False   # login itself doesn't need prior auth
        )
        # The session cookie is now stored in requests.Session() automatically
        return User.from_dict(response["user"])

class BaseClient:
    @property
    def is_authenticated(self) -> bool:
        # Just checks if the cookie exists in the session
        return "polimillionaire_auth" in self._session.cookies
  • requests.Session() automatically stores and sends cookies with every subsequent request — just like a browser session. Login once, all future requests carry the auth cookie automatically.No need to manually add auth headers to every request. The session handles it transparently.
  • The login endpoint itself doesn't need prior authentication (you can't log in if you need to be logged in first). All other endpoints default to auth_required=True.Guards against accidentally calling protected endpoints without logging in first.

Error Handling

millionaire_client/base.py — response handling
def _handle_response(self, response: requests.Response) -> dict:
    """Translate HTTP status codes → typed Python exceptions."""
    if response.status_code in (200, 201, 204):
        return response.json() if response.text else {}
    
    message = data.get("message", f"HTTP {response.status_code}")
    
    if response.status_code == 401:  raise AuthenticationError(message)
    if response.status_code == 404:  raise NotFoundError(message)
    if response.status_code == 400:  raise ValidationError(message)
    if response.status_code == 429:  raise RateLimitError(message)
    if response.status_code >= 500:  raise ServerError(message)
    raise MillionaireError(message)

Typed exceptions mean specific error handling. Instead of catching a generic requests.HTTPError and inspecting status codes everywhere, callers can except AuthenticationError or except RateLimitError specifically. The runner catches all exceptions broadly to prevent a game-crashing bug.

Section 04

Data Models — The Shape of Game Data

Think of these as official government forms. Every piece of information that passes between the server and the bot has a defined shape: exactly these fields, these types. No surprises. The models convert raw JSON dictionaries into typed Python objects so you get autocomplete and type safety throughout the codebase.

The Model Hierarchy

The Key Models Explained

GameState
Complete snapshot of a game in progress: current level, earned amount, active question, time remaining, money pyramid.
Question
One quiz question: text + 4 options (each with an ID and text). The option ID is needed for submitting answers.
AnswerResult
Server's response after submitting: correct/wrong/None, game_over, new earned_amount, next question if continuing.
GameStatus
Enum: IN_PROGRESS, COMPLETED (won), FAILED (wrong answer), TIMEOUT (30s expired).
millionaire_client/models.py — key fields explained
@dataclass
class AnswerResult:
    correct: Optional[bool] = None
    # None = server doesn't reveal (unusual)
    # True = correct answer, continue
    # False = wrong answer, game over

    game_over: bool = False     # True on both wrong answer AND final level
    earned_amount: float = 0    # Total prize accumulated so far
    timed_out: bool = False     # True if 30s deadline was exceeded
    
    reached_level: Optional[int] = None   # The level you "secured"
    current_level: Optional[int] = None   # Level of the just-answered question
    
    question: Optional[Question] = None   # Next question (if game continues)
    money_pyramid: List[MoneyLevel] = ... # Updated prize pyramid

The Level 0 Bug: The server sometimes omits the level field in the per-question payload, causing Question.from_dict to default it to 0. The GameAdapter detects this and injects the session's current_level as the correct value. Without this fix, every question would silently route to the "easy tier" (levels 1-5) in TieredStrategy.

Section 05

The Adapter Pattern — Anti-Corruption Layer

Imagine a power adapter when traveling internationally. Your laptop (the strategy) has its own plug shape. The foreign socket (API client) has a different shape. The adapter converts between them without changing either side. If the server API changes (different socket), you only replace the adapter — your laptop is unchanged.

The GameAdapter does three things:

  1. Translates API types (millionaire_client.Question) → domain types (GameQuestion)
  2. Exposes a clean interface for the runner (no HTTP knowledge needed)
  3. Fixes known server quirks (the level=0 bug)
polimibot/game/adapter.py — the translation layer
def _to_game_question(q: Optional[ApiQuestion]) -> Optional[GameQuestion]:
    """API question → our frozen DTO. None passes through (game over)."""
    if q is None:
        return None
    return GameQuestion(
        text=q.text,
        options=tuple(opt.text for opt in q.options),  # extract text only
        level=q.level,
    )

class GameAdapter:
    """Thin wrapper over MillionaireClient. One adapter == one game."""

    def __init__(self, client: MillionaireClient, competition_id: int) -> None:
        self._session = client.game.start(competition_id=competition_id)
        # ...

    @property
    def current_question(self) -> Optional[GameQuestion]:
        q = _to_game_question(self._session.current_question)
        if q is None:
            return None
        if q.level == 0:
            # Server quirk: level sometimes omitted → injected from session
            q = GameQuestion(text=q.text, options=q.options,
                           level=self._session.current_level)
        return q

    def submit_answer(self, option_index: int) -> AnswerOutcome:
        """Submit by 0-indexed option. Returns a typed outcome."""
        options = self._session.current_question.options
        option_id = options[option_index].id   # convert index → server ID
        result = self._session.answer(option_id=option_id)
        return AnswerOutcome(
            correct=result.correct,
            timed_out=result.timed_out,
            game_over=result.game_over,
            earned_amount=result.earned_amount,
            ...
        )
  • tuple(opt.text for opt in q.options) — the strategy only needs the text of each option, not the server-side option ID. Options are stored as plain strings.Strategies work with clean domain objects. They never need to know about option IDs.
  • The runner submits option_index (0, 1, 2, or 3). The adapter looks up options[option_index].id and sends that to the server.The rest of the system uses 0-based indices. The server uses integer IDs. Translation happens here, once.
  • The return type is AnswerOutcome (our domain type), not AnswerResult (the API type). The runner never touches AnswerResult.If the server adds new fields to AnswerResult, only the adapter needs updating — the runner is unaffected.

Section 06

The Game Loop — Runner Orchestrates Everything

The runner is the show producer behind the scenes. The host (server) sends questions. The contestant (strategy) answers. The producer (runner) coordinates everything: starts the show, starts the clock, cuts the contestant off if they run over time, logs everything, enforces the rate limit so the server isn't hammered, and wraps up at the end.

polimibot/runner.py — play_game() simplified
def play_game(client, competition_id, strategy, *,
              logger=None, fallback_index=0, verbose=True) -> GameResult:
    """Play one full game. The spine of the system."""

    # 1. Setup
    category = _category_for(competition_id)   # MATHS, HISTORY, etc.
    game = GameAdapter(client, competition_id)  # start the game session
    pacer = _Pacer(min_delay=1.5)               # rate limiter
    log = logger or RunLogger(...)               # JSONL file logger

    while not game.is_over:
        q = game.current_question
        if q is None:
            break

        # 2. Calculate time budget
        time_left = game.time_remaining_seconds  # from server deadline
        budget = min(time_left - 2.0, RUNTIME.hard_cutoff_seconds)

        # 3. Build StrategyInput (what the strategy sees)
        inp = StrategyInput(
            question=q.text,
            options=q.options,
            level=q.level,
            category=category,
            time_budget_seconds=budget,
        )

        # 4. Run strategy under watchdog (time-limited thread)
        try:
            out = _call_with_deadline(strategy, inp, budget)
        except _DeadlineExceeded:
            out = None    # timed out → use fallback

        # 5. Determine chosen index (abstain/timeout → fallback)
        chosen_idx = (out.chosen_index
                      if out and not out.is_abstain
                      else fallback_index)

        # 6. Submit (rate-limited)
        pacer.wait()
        outcome = game.submit_answer(chosen_idx)

        # 7. Learning loop (IndexGrower)
        grower = getattr(strategy, "index_grower", None)
        if grower:
            q_id = f"lvl_{q.level}"
            if outcome.correct:  grower.confirm(q_id)
            else:                grower.discard(q_id)

        # 8. Log everything
        log.log_question(QuestionRecord(...))

        if outcome.game_over:
            break

    # 9. End of game: flush learning, save summary
    if grower:
        grower.flush()
    log.log_summary(GameSummaryRecord(...))

    return GameResult(accuracy=n_correct/n_total, ...)
  • The category is looked up once before the loop — it's the same for every question in a competition. This routing hint flows all the way into the strategy's retrieval filter.Avoids repeated dictionary lookups; category is passed on every StrategyInput.
  • Time budget = min(server time remaining - 2s, hard cutoff). The 2-second margin accounts for network latency on submission. The hard cutoff (25s) is our own safety cap.If the server says 28s remaining, we give the strategy 26s. The extra 2s ensures the answer always arrives before the server deadline.
  • out.is_abstain — if the strategy couldn't produce a confident answer (parse failure, all tools abstained), it sets this flag. The runner then submits fallback_index=0 (option A by default).Honest abstention prevents strategies from silently defaulting to "A" without the caller knowing. The fallback is explicit and controllable.
  • Rate-limiter enforces 1.5 seconds between API calls. The game server is a proof-of-concept system — rapid-fire requests could overload it.Politeness: be a good API citizen.
  • After EVERY question: confirm correct answers (adds to index permanently), discard wrong ones (prevents pollution). Only then log.The learning loop must run AFTER the server confirms correctness but BEFORE the log, so the log reflects the final learned state.
Section 07

Watchdog & Pacer — Time Safety Mechanisms

The Watchdog is a kitchen timer that can't be ignored. You set it for 25 seconds. When it rings, the chef (strategy) MUST stop, whether the dish is ready or not. The Pacer is the pause between dishes — you don't serve the next course immediately after the last one; you give the diner (server) a moment to breathe.

The Watchdog — Deadline Enforcement

polimibot/runner.py — _call_with_deadline()
def _call_with_deadline(
    strategy: Strategy, inp: StrategyInput, deadline_seconds: float
) -> StrategyOutput:
    """Run strategy in a daemon thread. Raise _DeadlineExceeded on overrun."""
    box: dict = {}

    def target():
        try:
            box["out"] = strategy.answer(inp)
        except BaseException as exc:
            box["err"] = exc

    # Run in a daemon thread so it's killed when the main process exits
    th = threading.Thread(target=target, daemon=True)
    th.start()
    th.join(deadline_seconds)   # wait at most deadline_seconds

    if th.is_alive():
        # Thread still running → DEADLINE EXCEEDED
        raise _DeadlineExceeded(f"strategy exceeded {deadline_seconds:.1f}s")
        # Note: the thread continues running in background (can't kill it safely)
        # but its result is ignored — we've already moved on

    if "err" in box:
        raise box["err"]   # re-raise any strategy exception
    return box["out"]
  • The strategy runs in a daemon thread. Python can't safely interrupt a thread mid-CUDA-call (that would corrupt GPU state), so we don't kill it — we just stop waiting for it.The thread continues in the background (burning some GPU) but its result is irrelevant — we've already submitted the fallback answer.
  • th.join(deadline_seconds) blocks the main thread for at most that many seconds. If the thread finishes before the deadline → returns immediately. If not → continues past the join (thread still alive).This is the only safe way to impose a time limit on a Python thread that may be doing GPU computation.
  • Daemon threads are automatically killed when the main process exits. If the main program crashes or exits, the background thread doesn't keep the process alive.No zombie processes from hung GPU calls.

The Pacer — Rate Limiter

polimibot/runner.py — _Pacer
class _Pacer:
    """Ensures at least `min_delay` seconds between waits()."""

    def __init__(self, min_delay: float) -> None:
        self._min_delay = min_delay   # 1.5 seconds by default
        self._last = 0.0              # timestamp of last call

    def wait(self) -> None:
        elapsed = time.monotonic() - self._last
        if 0 < elapsed < self._min_delay:
            time.sleep(self._min_delay - elapsed)   # sleep the remaining gap
        self._last = time.monotonic()

# In the game loop:
pacer.wait()                        # called before EVERY submit
outcome = game.submit_answer(idx)   # then submit
  • time.monotonic() is used instead of time.time(). Monotonic time never goes backward (unlike wall-clock time which can be adjusted by NTP). This prevents the rare bug where a time.time() adjustment makes the elapsed time negative, causing a negative sleep.Correctness: even if the system clock is adjusted, rate limiting still works correctly.
  • The pacer only sleeps for the remaining gap. If the LLM already took 2.3 seconds (longer than the 1.5s minimum), no sleep is needed — we can submit immediately.Never adds unnecessary delay. Respects the minimum while not wasting time.

What Happens on a Timeout?

Timeout flow in the runner
# The strategy takes too long (e.g., LLM CUDA compilation stalls)
try:
    out = _call_with_deadline(strategy, inp, budget=23.0)
    # Strategy thread is still running at 23.0 seconds...
except _DeadlineExceeded:
    out = None

    timed_out=True,
    chosen_index=0,
    ...

What Happens on a Strategy Error?

Exception handling in the runner
try:
    out = _call_with_deadline(strategy, inp, budget)
except _DeadlineExceeded:
    out = None    # timed out
except Exception as exc:
    out = None    # strategy crashed
    print(f"! strategy raised {type(exc).__name__}: {exc}")

# In BOTH cases: out=None → submit fallback, continue the game
# The game never crashes because of a strategy bug

The runner is the safety net. No strategy bug can crash a live game. Timeouts, exceptions, parse failures, CUDA errors — all are caught by the runner and handled gracefully by submitting the fallback answer. The game continues. All errors are logged. This is intentional: the cost of a wrong answer (fallback) is much lower than crashing mid-game and losing all accumulated prize money.
