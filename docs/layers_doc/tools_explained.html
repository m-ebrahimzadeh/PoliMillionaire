Section 1

Why Tools? — The LLM's Blind Spot

Imagine a brilliant literature professor trying to do your taxes. They can discuss Dostoevsky for hours but will mess up "What is 15% of $47,200?" every time. Not because they're dumb — their training wasn't designed for precise arithmetic. LLMs have the same problem: they're language machines, not calculators. When you need 2+2=4 (not "approximately 4"), you need a tool.

LLMs fail on arithmetic because:

ProblemWhy LLMs failTool solution
Multi-digit multiplicationTokens aren't numbers — "347" is processed as charactersPython computes 347 * 29 exactly
PercentagesLLMs sometimes hallucinate intermediate steps15/100 * 200 = 30.0 — always correct
Factorials / combinatoricsTraining data has limited examples above small nmath.factorial(12) = 479001600
Precision mattersLLM might say "about 30" when answer is exactly 30Deterministic: exact float/int output

Design philosophy: Precision over Recall. The tool only answers when it's 100% sure. If it can't parse the question into an expression, or the result doesn't match any option → it abstains and lets the LLM try. A wrong deterministic answer is worse than a wrong LLM guess (same outcome, but damages calibration because it was reported with 99% confidence).

Section 2

The Tool Interface — Two Methods, That's It

Think of each tool as a specialist doctor in an ER. The triage nurse (ToolStrategy) shows each patient's chart to each specialist in order. The specialist glances at it and says either "This is my case, I'll handle it" (can_handle → True, use → StrategyOutput) or "Not my specialty, pass" (can_handle → False or use → None). The patient moves to the next specialist or to the general practitioner (fallback LLM).

polimibot/tools/base.py — The entire Tool contract
class Tool(ABC):
    """A tool either answers confidently or abstains (returns None).

    Abstaining is correct behavior — it lets the next handler try.
    Never return a guess; only return when confidence is high.
    """
    name: str = "tool"

    @abstractmethod
    def can_handle(self, inp: StrategyInput) -> bool:
        """Fast guard based on metadata (category, level, etc.).
        No computation here — this must be instant."""

    @abstractmethod
    def use(self, inp: StrategyInput) -> Optional[StrategyOutput]:
        """Attempt to answer. Return None to abstain and pass control downstream."""
  • can_handle() is a fast gate — it checks metadata only (category, level). No parsing, no computation. It must run in microseconds because it's called for EVERY question.Don't waste time running a calculator on "Who wrote Hamlet?" — category check eliminates it instantly.
  • use() does the actual work. It returns None to abstain (couldn't solve) or a StrategyOutput with the answer. Never guesses — only returns when certain.The abstention contract is what makes the chain work. Each tool honestly reports its limitations.
  • The return type is Optional[StrategyOutput] — None is a first-class result meaning "I tried and couldn't do it."This separates "I can't handle this type of question" (can_handle=False) from "I can handle this type but this specific question is too complex" (use() returns None).

Two levels of filtering: can_handle filters by question TYPE (fast, zero-cost). use filters by question CONTENT (may involve parsing). This two-stage design prevents expensive parsing on questions that obviously aren't relevant.

Section 3

The Safe Evaluator — A Sandboxed Calculator

Imagine building a calculator for a children's classroom. You want them to compute 2+2 and sqrt(16), but you absolutely do NOT want them to type import os; os.system("format C:"). The safe evaluator is that childproof calculator — it only allows whitelisted math operations and blocks everything else at the AST (Abstract Syntax Tree) level.

Why Not Just Use eval()?

Python's eval() is a remote code execution vulnerability. The LLM's output feeds into this evaluator. If you use eval(), a prompt injection or malformed output like __import__('os').system('rm -rf /') would execute arbitrary system commands. The AST sandbox makes this physically impossible — disallowed node types are rejected before any code runs.

polimibot/tools/calculator.py — safe_eval()
"""Safe arithmetic evaluator using Python's AST — no exec(), no subprocess."""

# Whitelisted binary operators ONLY
_BINARY_OPS = {
    ast.Add: op.add,   ast.Sub: op.sub,
    ast.Mult: op.mul,  ast.Div: op.truediv,
    ast.Mod: op.mod,   ast.Pow: op.pow,
    ast.FloorDiv: op.floordiv,
}

_UNARY_OPS = {
    ast.USub: op.neg,  ast.UAdd: op.pos,
}

# Whitelisted names and functions — expand carefully
_ALLOWED = {
    "abs": abs, "round": round, "min": min, "max": max,
    "sqrt": math.sqrt, "factorial": math.factorial,
    "log": math.log, "log2": math.log2, "log10": math.log10,
    "sin": math.sin, "cos": math.cos, "tan": math.tan,
    "gcd": math.gcd, "comb": math.comb, "perm": math.perm,
    "pi": math.pi, "e": math.e,
}


def safe_eval(expression: str) -> int | float:
    """Evaluate a numeric Python expression in a restricted AST sandbox.

    >>> safe_eval("15 / 100 * 200")
    30.0
    >>> safe_eval("factorial(5)")
    120
    """
    tree = ast.parse(expression.strip(), mode="eval")
    return _eval_node(tree.body)

The AST Walker

polimibot/tools/calculator.py — _eval_node()
def _eval_node(node: ast.AST) -> Any:
    # Numbers: 42, 3.14
    if isinstance(node, ast.Constant):
        if isinstance(node.value, (int, float)):
            return node.value
        raise ValueError(f"Non-numeric constant: {node.value!r}")

    # Binary ops: 2 + 3, 10 ** 2
    if isinstance(node, ast.BinOp):
        fn = _BINARY_OPS.get(type(node.op))
        if fn is None:
            raise ValueError(f"Operator {type(node.op).__name__} not allowed")
        return fn(_eval_node(node.left), _eval_node(node.right))

    # Unary ops: -5, +3
    if isinstance(node, ast.UnaryOp):
        fn = _UNARY_OPS.get(type(node.op))
        if fn is None:
            raise ValueError(f"Unary {type(node.op).__name__} not allowed")
        return fn(_eval_node(node.operand))

    # Function calls: sqrt(16), factorial(5)
    if isinstance(node, ast.Call):
        if not isinstance(node.func, ast.Name):
            raise ValueError("Only simple function names allowed")
        fn = _ALLOWED.get(node.func.id)
        if fn is None:
            raise ValueError(f"Function '{node.func.id}' not in whitelist")
        return fn(*[_eval_node(a) for a in node.args])

    # Named constants: pi, e
    if isinstance(node, ast.Name):
        if node.id in _ALLOWED:
            return _ALLOWED[node.id]
        raise ValueError(f"Name '{node.id}' not in whitelist")

    # ANYTHING ELSE → REJECTED
    raise ValueError(f"AST node {type(node).__name__} not allowed")
  • ast.parse(expression, mode="eval") converts the string into a tree structure WITHOUT executing anything. This is the safety key — we inspect the tree, not run the code.Parsing ≠ executing. We can analyze what the expression WOULD do without actually doing it.
  • Only int and float literals pass. Strings, booleans, None → rejected. No "hello", no True, no None.A calculator shouldn't process strings. Only numbers in, numbers out.
  • Only 7 operators allowed: +, -, *, /, %, **, //. No bitwise (&, |, ^), no shifts (<<, >>), no comparisons (<, >).Minimal surface area. Only what's needed for arithmetic.
  • Function calls must be simple names (no method calls like obj.method()). Only whitelisted functions execute.Prevents __import__('os').system('...') — method chaining and attribute access are blocked at the AST level.
  • Any AST node type not explicitly handled → ValueError. This is the crucial default: deny by default, allow only what's explicitly safe.New Python features or obscure syntax can't sneak through. Unknown = blocked.

MathsTool — From English to Numbers

Think of MathsTool as a bilingual translator + calculator. The question arrives in English ("What is 15% of 400?"). The translator converts it to math ("15/100*400"). The calculator evaluates it (= 60.0). Then the matcher checks: "Is 60 one of the options? Yes — it's option C!" The key insight: each step can fail, and failure means graceful abstention, not a wrong answer.

Step 2: Expression Extraction

polimibot/tools/maths_tool.py — _extract_expression()
# Pattern that signals "a computable expression follows"
_COMPUTE_PREFIX = re.compile(
    r'(?:what\s+is|calculate|compute|find\s+the\s+value\s+of|evaluate|simplify)'
    r'\s+(.+)',
    re.IGNORECASE,
)

def _extract_expression(question: str) -> Optional[str]:
    """Normalize and extract an arithmetic expression. Returns None if ambiguous."""
    q = _normalize(question)
    m = _COMPUTE_PREFIX.search(q)
    if not m:
        return None          # no "what is" / "calculate" → abstain
    fragment = m.group(1).strip().rstrip('?').strip()
    # Remove trailing prose ("...where x = 5")
    fragment = re.split(r'\bwhere\b|\bif\b|\bgiven\b', fragment)[0].strip()
    # Drop non-arithmetic characters
    cleaned = _KEEP.sub(' ', fragment).strip()
    return cleaned if cleaned else None
  • Only questions starting with "What is", "Calculate", "Compute", "Evaluate", "Simplify" are attempted. "Who invented calculus?" → no match → abstain.Conservative: don't try to parse non-computable questions. Better to miss a few than to get one wrong.
  • Strips trailing conditions like "where x = 5" or "if n = 10" — these are variable definitions that would need a different parser.Keep it simple: only handle self-contained arithmetic expressions.
  • Returns None at every failure point. The caller (MathsTool.use()) treats None as "abstain."Every step has an exit to safety. Ambiguity → don't guess.

Step 4: Option Matching

polimibot/tools/maths_tool.py — _match_options()
def _parse_option_value(text: str) -> Optional[float]:
    """Parse an option string as a float. "€1,000" → 1000.0, "30 kg" → 30.0"""
    cleaned = re.sub(r'[€$£,]', '', text)          # strip currency/commas
    cleaned = re.split(r'\s+[a-zA-Z]', cleaned)[0]  # "30 meters" → "30"
    try:
        return float(cleaned)
    except ValueError:
        return None


def _match_options(result: float, options: tuple[str, ...]) -> Optional[int]:
    """Return index of matching option, or None."""
    parsed = [_parse_option_value(opt) for opt in options]

    # Require at least 3 of 4 options to be numeric
    numeric_count = sum(1 for v in parsed if v is not None)
    if numeric_count < 3:
        return None   # not a numeric-options question

    # Float-safe comparison with relative tolerance
    tol = max(1e-9, abs(result) * 1e-9)
    for i, v in enumerate(parsed):
        if v is not None and abs(v - result) <= tol:
            return i
    return None   # computed value doesn't match any option
  • Options like "€1,000", "30 kg", "$4.50" are cleaned to pure numbers. Currency symbols, commas, and trailing units are stripped.Quiz options come in many formats. The parser handles the common ones without being fragile.
  • Requires at least 3 of 4 options to be numeric. This prevents false matches on questions like "Which scientist...?" where one option happens to contain a number.Safety heuristic: if most options aren't numbers, this isn't a numeric question.
  • Uses relative tolerance (1e-9) for float comparison. 0.1 + 0.2 == 0.3 is false in floating-point, but abs(0.30000000000000004 - 0.3) <= 1e-9 is true.Floating-point arithmetic has precision issues. Tolerance prevents false negatives from rounding.

The Complete MathsTool

polimibot/tools/maths_tool.py — MathsTool class
class MathsTool(Tool):
    """Deterministic arithmetic solver. Only answers when it can prove the answer.

    Coverage: ~30–50% of maths questions (those with computable answers
    and numeric options). The rest fall through to the LLM.
    """
    name = "maths_tool"

    def can_handle(self, inp: StrategyInput) -> bool:
        return inp.category == Category.MATHS   # instant check

    def use(self, inp: StrategyInput) -> Optional[StrategyOutput]:
        expr = _extract_expression(inp.question)
        if expr is None:
            return None          # can't parse → abstain

        try:
            result = safe_eval(expr)
        except Exception:
            return None          # math error → abstain

        if not isinstance(result, (int, float)) or not math.isfinite(result):
            return None          # NaN/Inf → abstain

        idx = _match_options(float(result), inp.options)
        if idx is None:
            return None          # doesn't match any option → abstain

        return StrategyOutput(
            chosen_index=idx,
            confidence=0.99,     # deterministic = near-certain
            rationale=f"MathsTool: {expr} = {result}",
            extras={"tool": "maths_tool", "expr": expr, "result": result},
        )

Count the exit points: There are 4 places where the tool returns None (abstains). Only ONE path leads to an answer. This is intentional — the tool has a narrow corridor of success and a wide highway of graceful failure. When it DOES answer, you can trust it.

Section 5

Symbolic & Statistical Tools — Beyond Arithmetic

MathsTool is a pocket calculator; some Maths questions need a graphing calculator and a stats table. "What is 15% of 400?" is arithmetic — MathsTool's safe_eval handles it. But "find x if 3x(x+5) = 2(3x+x+5)", "the remainder when 2^87 is divided by 7", or "the probability that at least 3 of 5 cyclones become hurricanes" need algebra, exact modular arithmetic, and probability distributions. Two more tools cover those — each following MathsTool's exact precision-over-recall contract: abstain on anything ambiguous, answer only when the result matches an option.

SympyDirectTool — Symbolic Math, Before the LLM

SympyDirectTool runs before the LLM (no agent loop, no LLM at any point — purely deterministic). It carries a library of 23 regex patterns (_PATTERNS) that recognise common Maths-question shapes, build a SymPy expression, evaluate it exactly, and check the result against the options. can_handle() gates it to the MATHS category; patterns are tried in order and the first that produces an option-matching result returns with confidence 0.97, otherwise the tool abstains. It sits after MathsTool in the chain: MathsTool → SympyDirectTool → BaselineLLMStrategy.

Equations / unknowns"If 3x(x+5) = 2(3x+x+5), find x"solve(...)
Modular arithmetic"remainder when 2^87 is divided by 7"Mod(2**87, 7)
Combinatorics"how many ways to choose 3 from 25"binomial(25, 3)

The engine: sympy_solve (polimibot/tools/sympy_tool.py). Both SympyDirectTool and the AgentStrategy ReAct loop evaluate symbolic expressions through the same sympy_solve function. It exposes only a curated, sandboxed subset of SymPy names via _ALLOWED_NAMES (solve, solveset, Mod, binomial, factorial, simplify, isprime, summation, …) — no file I/O, no os, no exec — and is lazy-imported (~2 s cold, <1 ms warm) so startup stays fast. An expression containing a single = is rewritten LHS - (RHS) and solved for its first free symbol; pure expressions evaluate to 15 significant figures.

Two extra guardrails because sympify() uses Python's eval() internally (a restricted namespace alone is not enough): a _BLOCKED regex rejects dunders, import, open(, exec(, eval(, and getattr/setattr/globals/locals( before parsing, and the result is rejected unless it is a genuine SymPy Basic — so an expression that escapes to a module or builtin never returns a value.

StatsTool — Probability Distributions

StatsTool covers question types neither MathsTool nor SympyDirectTool can: probability questions that need scipy.stats. Like the others it is MATHS-only, lazy-imports scipy on first call (~50 ms cold, <1 ms warm), and abstains immediately on any parse failure or ambiguous phrasing. Five ordered pattern functions cover binomial P(X≥k), repeated-success pk, binomial P(X=k), normal percentile, and normal sample-mean; a match returns with confidence 0.97. It sits last: MathsTool → SympyDirectTool → StatsTool → LLM fallback.

All three are Tool implementations, so they slot into the ToolStrategy chain (next section) in priority order: try the cheap deterministic solver first, fall through to the next, and only call the LLM if every tool abstains. Like a customer service escalation chain. Tier 1 (tool) tries to solve the problem. If they can't → escalate to Tier 2 (next tool). If nobody can → escalate to the manager (fallback LLM). The beauty: adding a new Tier 1.5 specialist requires ZERO changes to existing tiers — just insert them into the chain.

polimibot/strategies/tool_strategy.py
class ToolStrategy(Strategy):
    """Try registered tools in order; fall back to an LLM strategy.

    This is the Chain of Responsibility pattern:
    - "What" to try: the tool list (ordered by priority)
    - "How" to answer if tools don't help: the fallback strategy
    """

    def __init__(self, tools: list[Tool], fallback: Strategy) -> None:
        self.tools = tools
        self.fallback = fallback
        tool_names = "+".join(t.name for t in tools)
        self.name = f"tool[{tool_names}|fallback={fallback.name}]"

    def warm_up(self) -> None:
        """Warm up the fallback LLM (tools are instant-start)."""
        self.fallback.warm_up()

    def shutdown(self) -> None:
        self.fallback.shutdown()

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        for tool in self.tools:
            if tool.can_handle(inp):      # fast gate
                out = tool.use(inp)       # attempt
                if out is not None:
                    return out            # tool answered — done!
        # All tools abstained (or none could handle this input)
        return self.fallback.answer(inp)  # LLM takes over
  • for tool in self.tools — tools tried in insertion order. Put the most confident/fastest tool first. If MathsTool can answer, skip everything else.Priority chain: cheapest answer wins. No need to call the LLM if a tool already knows.
  • First can_handle (free), then use (may involve parsing). If can_handle is False, use() is never called.Avoid wasted computation: don't even try to parse a history question as math.
  • The fallback is itself a Strategy — could be BaselineLLM, RAGStrategy, or even another ToolStrategy. Arbitrary nesting.Composability: ToolStrategy(tools=[MathsTool()], fallback=RAGStrategy(...)) combines retrieval + computation.

Section 7

AgentStrategy — The Thinking Loop (ReAct)

Imagine a contestant on a game show who can whisper to a friend with a calculator. They read the question, think aloud ("Hmm, I need to find 2^10..."), whisper "What's 2 to the power of 10?" to their friend, get back "1024", think more ("So the answer is 1024, which matches option B"), and say their final answer. The AgentStrategy is this think→act→observe→conclude loop, automated.

The System Prompt

polimibot/strategies/agent_strategy.py — System message
_AGENT_SYSTEM = """\
You are a careful quiz contestant. You have one tool:

  CALL: calc()
  Evaluates a Python arithmetic expression and returns the numeric result.
  Example: CALL: calc(15/100 * 400)

Rules:
  1. Think step by step in at most 3 sentences.
  2. If computation is needed, emit EXACTLY one line: CALL: calc()
     Then STOP — you will receive the result.
  3. When you know the final answer, write EXACTLY: Answer: 
     where  is one of A, B, C, D.
  4. Never guess a number — use the tool if unsure.
"""
  • The system prompt defines a strict text protocol: CALL: calc(...) for tool use, Answer: X for the final answer. No JSON, no function-calling API — just text patterns.Works with any LLM that can follow text instructions. No model-specific function-calling features needed.
  • "Then STOP" tells the model to stop generating after the CALL line. This is reinforced by max_new_tokens=200 — tight budget prevents rambling.If the model kept generating after CALL, it would hallucinate the answer without waiting for the real result.
  • "Never guess a number — use the tool if unsure" — this instruction biases the model toward using the tool for any arithmetic rather than trying to compute in its head.Encourage tool use: the whole point is to offload computation to the deterministic calculator.

The Core Loop

polimibot/strategies/agent_strategy.py — answer()
def answer(self, inp: StrategyInput) -> StrategyOutput:
    deadline = time.monotonic() + inp.time_budget_seconds - 1.0  # 1s safety margin
    messages = self._initial_messages(inp)
    trace = []

    for iteration in range(self.max_iterations + 1):
        if time.monotonic() >= deadline:
            break                               # out of time!

        resp = self.llm.generate(messages, max_new_tokens=200, temperature=0.0)
        text = resp.text.strip()
        trace.append(f"[step {iteration}] {text}")
        messages.append({"role": "assistant", "content": text})

        # ── Branch 1: Tool call detected ──────────────────────────
        call = _extract_call(text)
        if call and time.monotonic() < deadline:
            tool_name, expression = call
            if tool_name == "calc":
                observation = _run_calc(expression)  # safe_eval
            else:
                observation = f"Unknown tool '{tool_name}'"
            # Inject result as a user message
            messages.append({
                "role": "user",
                "content": f"Tool result: {observation}\nNow give your final answer."
            })
            continue   # next iteration: model sees the result

        # ── Branch 2: Final answer detected ───────────────────────
        idx = parse_answer(text)
        if idx is not None:
            return StrategyOutput(chosen_index=idx, confidence=0.70)

        # ── Branch 3: Neither — nudge the model ──────────────────
        messages.append({
            "role": "user",
            "content": "Please write your final answer: 'Answer: '."
        })

    # ── Exhausted: salvage from trace ─────────────────────────────
    for step_text in reversed(trace):
        idx = parse_answer(step_text)
        if idx is not None:
            return StrategyOutput(chosen_index=idx, confidence=0.40)

    return StrategyOutput(chosen_index=0, is_abstain=True)  # give up
  • Hard deadline with 1-second safety margin. The game server times out at 30s — exceeding it = automatic wrong answer.Time-awareness: the agent won't spin forever. Safety margin accounts for network latency on submission.
  • If the LLM emits CALL: calc(...), execute it via safe_eval and inject the result as a new user message. Then loop again so the model can see the result.The model "observes" the tool's output as if another person told them the answer. Multi-turn simulation.
  • If the LLM emits Answer: B, we're done. Parse it and return immediately.Success path: the model reasoned (possibly with tool help) and committed to an answer.
  • If the output is neither a tool call nor a final answer (model is rambling), we nudge it: "Please write your final answer." One more chance.LLMs sometimes produce unfocused text. The nudge re-focuses them on the required output format.
  • If all iterations are exhausted, scan the trace backwards for any answer. Maybe the model said "Answer: C" in step 1 but then continued generating.Graceful degradation: extract whatever signal is available, even from imperfect outputs.

Tool Call Detection — Balanced Parentheses

polimibot/strategies/agent_strategy.py — _extract_call()
def _extract_call(text: str) -> Optional[tuple[str, str]]:
    """Find 'CALL: tool_name(expr)' with nested parens handled correctly.

    Returns (tool_name, expression) or None.
    Balanced-paren scan ensures 'calc(factorial(5))' extracts correctly.
    """
    m = re.search(r"CALL\s*:\s*(\w+)\s*\(", text, re.IGNORECASE)
    if not m:
        return None
    tool_name = m.group(1).lower()
    start = m.end()      # index of first char INSIDE the outer '('
    depth = 1
    for i, ch in enumerate(text[start:]):
        if ch == "(":
            depth += 1
        elif ch == ")":
            depth -= 1
            if depth == 0:
                return tool_name, text[start : start + i]
    return None   # unbalanced — malformed call

Why manual paren scanning instead of regex? Consider CALL: calc(factorial(5) + sqrt(16)). A regex like \((.+?)\) matches greedily to the first ) → captures only factorial(5 (broken!). A regex like \((.+)\) is greedy and might capture too much if there's text after the call. The manual depth-counting algorithm correctly finds the matching outer parenthesis regardless of nesting depth.

Security Design — Defense in Depth

Think of this as airport security with multiple checkpoints. Even if a passenger (LLM output) gets through one check, there are more layers behind it. The AST parser is the X-ray machine (inspects structure). The whitelist is the no-fly list (blocks known bad actors). The deny-by-default fallthrough is the armed guard at the gate (stops anything that slipped through).

When to Use Which

The TieredStrategy combines both: All maths questions first go through ToolStrategy (instant, deterministic). If the tool abstains (question too complex to parse), it falls back to AgentStrategy (LLM reasons step-by-step with tool access). This gives you the best of both worlds: instant answers when possible, reasoned answers when necessary.

Optimal composition for maths questions
# The tool answers ~40% of maths questions instantly (99% accuracy)
# The agent handles the rest with reasoning + computation (~70% accuracy)
# Together: better than either alone

maths_tool = MathsTool()
agent = AgentStrategy(llm, max_iterations=3)

# ToolStrategy tries MathsTool first, falls back to Agent
maths_solver = ToolStrategy(
    tools=[maths_tool],
    fallback=agent,
)

# In the TieredStrategy:
tiered = TieredStrategy(
    easy=baseline,
    medium=rag,
    hard=ensemble,
    maths_override=maths_solver,  # ALL maths goes here regardless of level
)

The architecture is designed for easy extension. To add a new tool:

    """Compute date differences, day-of-week, etc."""

        # Only try on questions mentioning dates/years

# Plug it in — zero changes to existing code:
multi_tool = ToolStrategy(
    tools=[MathsTool(), DateTool()],   # tried in order
    fallback=rag,
)

Open-Closed Principle in action: The ToolStrategy is open for extension (add new tools to the list) but closed for modification (never need to edit ToolStrategy, MathsTool, or any existing code). Each tool is a self-contained unit that plugs into the chain independently.
