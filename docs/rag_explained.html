Section 01

What is RAG?

RAG stands for Retrieval-Augmented Generation. Instead of relying purely on what a language model memorised during training, RAG lets the model look things up before answering.

💡
Simple Analogy Imagine you're on a quiz show and you're allowed to quickly flip through a mini-encyclopedia before answering each question. The AI does exactly this — it retrieves relevant Wikipedia passages and uses them as context when generating its answer.

In PoliMillionaire, the language model must answer multiple-choice trivia questions across four categories: Entertainment, History, Science, and Maths. Without RAG, the model relies entirely on its training data — which may be outdated or incomplete. With RAG, it can ground its answers in a curated, up-to-date knowledge base.

Section 02

Building the Knowledge Index

Before the game starts, notebook section 0.4 builds a searchable knowledge base from Wikipedia articles. This is a one-time process that produces the files the retriever queries during gameplay.

📥

Download Articles

Wikipedia articles for all four categories (Entertainment, History, Science, Maths) are fetched and stored in corpus.jsonl.

✂️

Chunk Text

Each article is split into overlapping 300-word passages with 50-word overlap, respecting sentence and section boundaries.

🧮

Embed Passages

Each chunk is converted into a 384-dimensional vector using BAAI/bge-small-en-v1.5 and stored in a FAISS index.

🔤

Build BM25

A parallel keyword index is built from the same chunks using BM25 — a probabilistic text-matching algorithm.

Section 03

Text Chunking — The Art of Splitting

How you split a Wikipedia article into pieces has a bigger impact on retrieval quality than which embedding model you use. The chunker in polimibot/rag/chunker.py makes several careful design choices.

Why Does Chunking Matter?

You can't embed an entire 5,000-word article — models have token limits, and long texts lose important detail in a single vector. But naive fixed-size splitting creates a different problem:

❌ Bad: Fixed-size character split
Chunk 1: "...Germany surrendered unconditionally on May 7,"
Chunk 2: "1945. The war in Europe was over. Japan continued"
→ The date "May 7, 1945" is split across chunks. Neither chunk has the complete fact.
✅ Good: Sentence-aware split (your code)
Chunk 1: "...Germany surrendered unconditionally on May 7, 1945. The war in Europe was over."
Chunk 2: "Japan continued fighting until August 15, 1945, when Emperor Hirohito announced..."
→ Complete sentences stay together. Facts are preserved.

The Overlap Trick

The last 50 words of Chunk 1 become the first 50 words of Chunk 2. This means a fact that falls near a boundary appears in both chunks — no information is lost at the seams.

Chunk 1: [word₁ … word₂₅₀] [word₂₅₁ … word₃₀₀]
↑ overlap zone (50 words) is shared ↑
Chunk 2:                           [word₂₅₁ … word₃₀₀] [word₃₀₁ … word₅₅₀]

The Four-Step Pipeline

1

Split into Sections

_split_sections() — Wikipedia-style == Header == lines become section boundaries. The header text is prepended to the first chunk of its section only, so the heading travels with its content.

2

Split into Sentences

_split_sentences() — Uses a regex to split on ., !, ? followed by whitespace. Conservative — won't perfectly handle "Mr. Smith" but a stray split just creates a short chunk that gets merged in step 4.

3

Greedy Sentence Packing

_pack_sentences() — Greedily fills a buffer up to 300 words by adding whole sentences. When the next sentence would overflow, the buffer is flushed as a chunk and the last 50 words are carried over as overlap.

4

Tail-Merge Short Chunks

If the final chunk of a document is fewer than chunk_size // 5 = 60 words, it's merged into its predecessor. A tiny 5-word chunk would produce a noisy embedding that competes unfairly for top-K slots.

polimibot/rag/chunker.py — core Chunk dataclass 
@dataclass(frozen=True)
class Chunk:
    """Atomic retrieval unit. Immutable so it's safe to cache and share."""
    text:     str            # the passage text shown to the LLM
    source:   str            # document title (shown in the prompt as citation)
    chunk_id: int            # 0-based position within that document
    category: Optional[str] = None  # "history", "science", etc. — used for filtering
Section 04

Embeddings — Fingerprints for Meaning

An embedding is a fixed-size vector of numbers that captures the semantic meaning of a piece of text. Similar meanings produce vectors that point in the same direction.

Your model, BAAI/bge-small-en-v1.5, converts any text into a vector of 384 numbers. Each number represents some abstract learned concept — nobody designs what each dimension means; the model learns during training on millions of text pairs.

Conceptual illustration — similar meanings → similar vectors 
# Texts with similar meaning produce vectors close together
"Who invented the telephone?"  →  [0.023, -0.157,  0.890, ...]  # Vector A
"Who created the phone?"       →  [0.031, -0.142,  0.882, ...]  # Vector B  ← very close!
"What's the weather today?"    →  [0.912,  0.231, -0.451, ...]  # Vector C  ← far away

similarity(A, B) = 0.96  # almost identical meaning!
similarity(A, C) = 0.12  # completely different topic

The Asymmetry Trick BGE Design

Your embedder.py contains a subtle but critical detail — queries and passages are encoded with different prefixes:

polimibot/rag/embedder.py — asymmetric query/passage encoding 
@dataclass(frozen=True)
class EmbedderSpec:
    model_name:     str = "BAAI/bge-small-en-v1.5"   # 384-dim, ~130 MB, CPU-friendly
    normalize:      bool = True    # L2-normalize → cosine sim becomes dot product
    query_prefix:   str = "Represent this sentence for searching relevant passages: "
    passage_prefix: str = ""       # passages get NO prefix

class Embedder:
    def encode_query(self, texts: list[str]) -> np.ndarray:
        """Embed retrieval queries — applies the query prefix."""
        return self._encode(texts, prefix=self.spec.query_prefix)

    def encode_passage(self, texts: list[str]) -> np.ndarray:
        """Embed corpus passages — NO prefix."""
        return self._encode(texts, prefix=self.spec.passage_prefix)
🎯
Why different prefixes? The BGE model was trained with a query-side instruction prompt to signal "I'm a question looking for answers." The prefix subtly rotates the query vector into the right part of the embedding space to better match passage vectors. Using the wrong prefix silently degrades cosine scores by 5–15%. The manifest system enforces consistency between index-build time and retrieval time.

L2 Normalisation

With normalize=True, every vector is scaled to have length exactly 1 (unit vector). This means the dot product between two vectors equals the cosine similarity — they measure the same angle between vectors regardless of magnitude. FAISS's IndexFlatIP (Inner Product) becomes equivalent to cosine similarity search.

Section 05

Search: BM25 + FAISS + RRF Fusion

Both a keyword index (BM25) and a semantic vector index (FAISS) are searched in parallel. Their ranked results are then merged using Reciprocal Rank Fusion into a single, better-ranked list.

BM25 — Keyword Search Sparse

BM25 is the gold standard for keyword search — think of it as Ctrl+F on steroids. It tokenizes the query and passage into words (applying stemming so "invented" and "inventing" match), then scores passages based on:

  • Term frequency (TF): More occurrences of the query word in the passage = higher score
  • Inverse document frequency (IDF): Rare words across the whole corpus are more discriminative = boosted
  • Document length normalization: Long documents aren't unfairly rewarded for containing more words

FAISS — Semantic Search Dense

FAISS stores all 384-dimensional chunk embeddings in a flat index. At search time, the query is embedded and its dot product is computed against every stored vector. The top-K nearest vectors correspond to the most semantically similar passages.

Your project uses IndexFlatIP — flat (exact, brute-force) inner product. No approximation, no hashing — every single vector is checked. This is fine for thousands of chunks and ensures no relevant passage is ever accidentally missed.

Reciprocal Rank Fusion (RRF) The Combiner

Here's the problem: BM25 scores live in [0, 30+] while dense scores live in [-1, +1]. You cannot simply sum them — BM25 would dominate. Instead, RRF ignores the scores entirely and uses only the rank positions:

RRF Formula (Cormack, Clarke, Buettcher 2009)
RRF(d) = Σᵢ 1 / (k + rankᵢ(d))
where k=60 (empirical constant), rankᵢ(d) is 1-indexed rank of document d in list i (∞ if absent)
Worked example — query: "quantum mechanics" 
# BM25 ranking (by keyword score):          # Dense ranking (by cosine score):
# 1. "Quantum theory article" (28.5)         # 1. "Physics article" (0.89)
# 2. "Physics article"       (12.3)         # 2. "Modern physics"  (0.82)
# 3. "Mechanics article"     ( 8.1)         # 3. "Quantum theory"  (0.77)

# RRF scores (k=60):
# "Quantum theory": 1/(60+1) + 1/(60+3)  = 0.01639 + 0.01587 = 0.03226
# "Physics":        1/(60+2) + 1/(60+1)  = 0.01613 + 0.01639 = 0.03252  ← WINNER!
# "Modern physics": 0        + 1/(60+2)  = 0.00000 + 0.01613 = 0.01613
# "Mechanics":      1/(60+3) + 0         = 0.01587 + 0.00000 = 0.01587

# "Physics" wins because it ranked HIGH IN BOTH lists, even though it was
# #1 in neither. RRF captures consensus across retrieval methods.

Source Diversification

After fusion, a pass of source-aware top-K selection ensures the final K passages come from different Wikipedia articles where possible. If 5 of the top 7 results all come from the same article, the retriever picks the best from each unique source first, then backfills from remaining candidates.

Not every question has good coverage in the offline index. Feeding low-quality passages to the LLM is worse than no context at all — it introduces noise. The score gate catches this, and the live search fallback provides a safety net.

When the offline index fails the gate, LiveSearchFallback fires a real-time Wikipedia API query:

polimibot/rag/live_search.py — simplified flow 
# 1. Search Wikipedia for article titles matching the question
article_titles = wikipedia_api.search("Who invented the telephone?")
# → ["Alexander Graham Bell", "Invention of the telephone", ...]

# 2. Fetch the top article's full text
article_text = wikipedia_api.get_page("Alexander Graham Bell")

# 3. Chunk it on-the-fly (same 300-word chunks as the offline index)
chunks = chunk_text(article_text, source="Alexander_Graham_Bell", chunk_size=300, overlap=50)

# 4. Search those chunks with BM25 (no time to embed for FAISS)
top_chunks = bm25_in_memory_search(query, chunks, k=3)

# 5. These chunks are returned AND buffered in IndexGrower for future sessions
index_grower.buffer(article_text, question_id="Q_level_5_42")
🌱
Learning Happens Here Every successful live search feeds into the IndexGrower. If the answer is confirmed correct, the fetched article is permanently added to the offline index. Next time a similar question appears, the offline retrieval succeeds — no live search needed!

The two-stage pipeline gets you cross-encoder accuracy at near bi-encoder speed: the bi-encoder retrieves a broad candidate set of ~15 passages, then the cross-encoder precisely reranks just those 15 — 15 forward passes instead of 10,000.

polimibot/rag/reranker.py — reranking core 
def rerank(self, query: str, candidates, *, top_k=None):
    """
    Re-score retrieval candidates by joint query-chunk relevance.
    The original dense/RRF score is DISCARDED — only chunk text matters now.
    """
    # Build (query, passage_text) pairs for joint encoding
    pairs = [(query, chunk.text) for chunk, _ in candidates]

    # Score all pairs — one model forward pass per pair (the expensive part)
    scores = self._score_pairs(pairs)   # uses BAAI/bge-reranker-base

    # Re-sort by cross-encoder score (descending)
    scored = [(chunk, float(s)) for (chunk, _), s in zip(candidates, scores)]
    scored.sort(key=lambda x: x[1], reverse=True)

    return scored if top_k is None else scored[:top_k]
Section 08

LLM Answer Scoring

Once relevant passages are retrieved and formatted as context, there are two ways to extract the LLM's answer: by measuring option probabilities directly (logit scoring) or by letting the model freely generate its reasoning.

The Formatted Prompt

Passages are numbered, truncated at sentence boundaries, and decorated with source citations. The prompt block looks like this:

rag_strategy.py — example formatted context 
"""
Reference material (retrieved from Wikipedia):

[1] From "World War II" (Article: World_War_II):
World War II ended on September 2, 1945, when Japan formally surrendered
aboard the USS Missouri in Tokyo Bay. The war in Europe had concluded in
May 1945 with Germany's surrender...

[2] From "Surrender of Japan" (Article: Surrender_of_Japan):
The surrender of Japan marked the end of World War II. Emperor Hirohito
announced Japan's acceptance of the Potsdam Declaration on August 15, 1945...

[3] From "Victory over Japan Day" (Article: Victory_over_Japan_Day):
VJ Day celebrations occurred after 1945 when the Allies achieved victory...

---

Question: What year did World War II end?
A) 1943  B) 1945  C) 1947  D) 1950
"""

Two configuration limits control the context size: RAG_MAX_PASSAGE_CHARS = 800 per passage, and RAG_MAX_TOTAL_CHARS = 2400 total. Truncation always happens at sentence boundaries — the _truncate_to_sentence() helper finds the last . / ! / ? before the character limit.

Mode A: Logit Scoring RAG_USE_SCORE_OPTIONS=True

Instead of generating text, the LLM is presented with the full prompt ending in "Answer:" and we measure the probability it assigns to each option letter token at that position:

Conceptual illustration — logit scoring 
prompt = f"{formatted_context}\n\nAnswer:"

# The model computes a probability distribution over its entire vocabulary.
# We extract the probabilities for exactly the tokens "A", "B", "C", "D":
P(next_token = "A") = 0.05
P(next_token = "B") = 0.89  # ← highest!
P(next_token = "C") = 0.03
P(next_token = "D") = 0.03

answer = "B"   # pick the letter with the highest logit / probability
⚡
Why Logit Scoring is Preferred It requires exactly one forward pass and produces a clean, unambiguous answer. Free generation might say "I believe it's B" or "The answer is 1945" — requiring regex parsing that can fail. Logit scoring also exposes calibrated confidence values useful for downstream ensemble strategies.

Mode B: Free Generation RAG_USE_SCORE_OPTIONS=False

The LLM generates a free-text explanation (chain-of-thought or elimination reasoning), which is then parsed to extract the chosen option letter. This mode supports ZERO_SHOT_COT, FEW_SHOT_COT, and ELIMINATION prompt styles from polimibot/prompts/templates.py.

Section 09

The Self-Growing Index (IndexGrower)

The most distinctive feature of this RAG system: the knowledge index literally grows smarter during a game session by learning from successful live searches. The IndexGrower in polimibot/rag/index_grower.py manages this learning loop.

🧬
Core Design Principle Only articles that led to correct answers are added to the permanent index. If the live search retrieved misleading information and the bot answered wrong, those chunks are silently discarded. This prevents the index from being polluted with harmful content.

The Three-Phase Lifecycle

📥

Phase 1: Buffer

When live search fetches a new Wikipedia article, index_grower.buffer(article, question_id) is called. The article is chunked and stored in a pending buffer keyed by the unique question ID. These chunks are immediately available for answering the current question but are NOT yet in the permanent index.

✅

Phase 2a: Confirm (correct answer)

index_grower.confirm(question_id) is called by runner.py after the game server confirms the answer was correct. The pending chunks for that question are embedded using the full FAISS embedder and appended to the in-memory FAISS and BM25 indices. Future questions that need this information will now find it in the offline index.

🗑️

Phase 2b: Discard (wrong answer)

If the answer was wrong, the pending buffer for that question ID is quietly dropped. The article that may have contributed to the mistake never enters the index.

💾

Phase 3: Flush (end of session)

index_grower.flush() persists all in-memory growth to disk: the FAISS .faiss file is re-saved, the BM25 .json is re-saved, and the raw article texts are appended to corpus.jsonl. Next game session starts with a richer index.

polimibot/runner.py — IndexGrower wiring after each answer 
# After submitting the answer and receiving the game server's verdict:
if hasattr(strategy, 'index_grower') and strategy.index_grower:
    grower = strategy.index_grower
    if answer_was_correct:
        # Embed and permanently add buffered chunks to the live index
        grower.confirm(question_id)
    else:
        # Discard — don't learn from potentially misleading content
        grower.discard(question_id)

# At the very end of the game session:
if hasattr(strategy, 'index_grower') and strategy.index_grower:
    strategy.index_grower.flush()   # persist to disk
