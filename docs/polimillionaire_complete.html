Introduction

What is PoliMillionaire?

PoliMillionaire is "Who Wants to Be a Millionaire?" — but the contestant is an AI bot. A server asks multiple-choice questions. The bot has 30 seconds to answer each one. Correct → advance to the next level and earn more money. Wrong → game over (but you keep the "safe amount" from milestone levels). Reach level 15 → maximum prize of €1,000,000.

This project builds a sophisticated AI system that combines multiple techniques — large language models, retrieval-augmented generation, deterministic computation tools, and ensemble strategies — to answer trivia questions across six categories with high accuracy.

The Money Pyramid

Each level corresponds to a prize amount. Getting a question wrong drops you back to the last "milestone" (safe amount). The pyramid structure creates increasing stakes:

Architecture

System Architecture — Four Clean Layers

Imagine a phone interpreter at a diplomatic summit. The diplomat (Strategy) speaks only English and doesn't know foreign protocol. The interpreter (Adapter) translates between English and the foreign language. The phone (API Client) handles actual transmission. Separating these three concerns means you can change any one without touching the others.

The Adapter Pattern enforces a strict boundary. The rule: "Anywhere outside adapter.py importing millionaire_client is a code smell." All 30+ strategies, the evaluator, the logging system — none of them ever touch the API client directly. Benefits: (1) test strategies with mock adapters; (2) if the API changes, fix only one file; (3) strategies stay clean and focused.

The HTTP Client — Talking to the Server

Think of it like a bank teller system. You (the application) give orders to the teller (MillionaireClient). The teller handles all the paperwork: authentication tokens, request formatting, error translation, and response parsing. You never deal with raw HTTP — just clean Python method calls.

Authentication: Cookie-Based Sessions

millionaire_client/auth.py + base.py — login flow
class AuthModule:
    def login(self, username: str, password: str) -> User:
        # POST /api/auth/login → server sets "polimillionaire_auth" cookie

            "/api/auth/login",
            data={"username": username, "password": password},
            auth_required=False   # login itself doesn't need prior auth
        )
        # The session cookie is now stored in requests.Session() automatically

class BaseClient:
    @property
    def is_authenticated(self) -> bool:
        return "polimillionaire_auth" in self._session.cookies
  • requests.Session() automatically stores and sends cookies with every subsequent request — just like a browser session. Login once, all future requests carry the auth cookie automatically.No need to manually add auth headers to every request.
  • The login endpoint itself doesn't need prior authentication. All other endpoints default to auth_required=True.Guards against accidentally calling protected endpoints without logging in first.

Error Handling: Typed Exceptions

millionaire_client/base.py — response handling
def _handle_response(self, response: requests.Response) -> dict:
    """Translate HTTP status codes → typed Python exceptions."""
    if response.status_code in (200, 201, 204):
        return response.json() if response.text else {}
    
    message = data.get("message", f"HTTP {response.status_code}")
    
    if response.status_code == 401:  raise AuthenticationError(message)
    if response.status_code == 404:  raise NotFoundError(message)
    if response.status_code == 400:  raise ValidationError(message)
    if response.status_code == 429:  raise RateLimitError(message)
    if response.status_code >= 500:  raise ServerError(message)
    raise MillionaireError(message)

Typed exceptions mean specific error handling. Instead of catching a generic requests.HTTPError and inspecting status codes everywhere, callers can except RateLimitError specifically. The runner catches all exceptions broadly to prevent game-crashing bugs.

The Adapter Pattern — Anti-Corruption Layer

Imagine a power adapter when traveling internationally. Your laptop (the strategy) has its own plug shape. The foreign socket (API client) has a different shape. The adapter converts between them without changing either side. If the server API changes (different socket), you only replace the adapter.

polimibot/game/adapter.py — the translation layer
class GameAdapter:
    """Thin wrapper over MillionaireClient. One adapter == one game."""

    def __init__(self, client: MillionaireClient, competition_id: int):
        self._session = client.game.start(competition_id=competition_id)

    @property
    def current_question(self) -> Optional[GameQuestion]:
        q = _to_game_question(self._session.current_question)
        if q is None:
            return None
        if q.level == 0:
            # Server quirk: level sometimes omitted → inject from session
            q = GameQuestion(text=q.text, options=q.options,
                           level=self._session.current_level)
        return q

    def submit_answer(self, option_index: int) -> AnswerOutcome:
        """Submit by 0-indexed option. Returns a typed outcome."""
        options = self._session.current_question.options
        option_id = options[option_index].id  # convert index → server ID
        result = self._session.answer(option_id=option_id)
        return AnswerOutcome(correct=result.correct, ...)
  • Options are stored as plain strings in GameQuestion. The strategy never needs server-side option IDs.Clean domain objects. Strategies work with text, not database IDs.
  • The runner submits option_index (0-3). The adapter looks up options[option_index].id and sends that to the server.Translation happens once, in one place.

The Game Loop — Runner Orchestrates Everything

The runner is the show producer behind the scenes. The host (server) sends questions. The contestant (strategy) answers. The producer (runner) coordinates everything: starts the show, starts the clock, cuts the contestant off if they run over time, logs everything, enforces the rate limit, and wraps up at the end.

polimibot/runner.py — play_game() simplified
def play_game(client, competition_id, strategy, *,
              logger=None, fallback_index=0) -> GameResult:
    """Play one full game. The spine of the system."""

    # 1. Setup
    category = _category_for(competition_id)
    game = GameAdapter(client, competition_id)
    pacer = _Pacer(min_delay=1.5)
    log = logger or RunLogger(...)

    while not game.is_over:
        q = game.current_question
        if q is None:
            break

        # 2. Calculate time budget
        time_left = game.time_remaining_seconds
        budget = min(time_left - 2.0, RUNTIME.hard_cutoff_seconds)

        # 3. Build StrategyInput
        inp = StrategyInput(
            question=q.text, options=q.options,
            level=q.level, category=category,
            time_budget_seconds=budget,
        )

        # 4. Run strategy under watchdog
        try:
            out = _call_with_deadline(strategy, inp, budget)
        except _DeadlineExceeded:
            out = None

        # 5. Determine chosen index
        chosen_idx = (out.chosen_index
                      if out and not out.is_abstain
                      else fallback_index)

        # 6. Submit (rate-limited)
        pacer.wait()
        outcome = game.submit_answer(chosen_idx)

        # 7. Learning loop (IndexGrower)
        grower = getattr(strategy, "index_grower", None)
        if grower:
            if outcome.correct: grower.confirm(q_id)
            else:               grower.discard(q_id)

        # 8. Log everything
        log.log_question(QuestionRecord(...))

    return GameResult(accuracy=n_correct/n_total, ...)
  • Category is looked up once before the loop — same for every question in a competition.Avoids repeated dictionary lookups.
  • Time budget = min(server time - 2s, hard cutoff). The 2-second margin accounts for network latency.Ensures the answer always arrives before the server deadline.
  • If the strategy sets is_abstain=True, the runner submits fallback_index=0 (option A).Honest abstention is explicit and controllable.
  • Rate-limiter enforces 1.5 seconds between API calls.Be a good API citizen — don't hammer the server.
  • After EVERY question: confirm correct answers (adds to index permanently), discard wrong ones.The learning loop runs AFTER server confirmation but BEFORE logging.

Watchdog & Pacer — Time Safety Mechanisms

The Watchdog is a kitchen timer that can't be ignored. You set it for 25 seconds. When it rings, the chef (strategy) MUST stop, whether the dish is ready or not. The Pacer is the pause between dishes — you don't serve the next course immediately after the last one.

polimibot/runner.py — _call_with_deadline()
def _call_with_deadline(strategy, inp, deadline_seconds):
    """Run strategy in a daemon thread. Raise _DeadlineExceeded on overrun."""
    box: dict = {}

    def target():
        try:
            box["out"] = strategy.answer(inp)
        except BaseException as exc:
            box["err"] = exc

    th = threading.Thread(target=target, daemon=True)
    th.start()
    th.join(deadline_seconds)  # wait at most deadline_seconds

    if th.is_alive():
        raise _DeadlineExceeded(f"exceeded {deadline_seconds:.1f}s")
    if "err" in box:
        raise box["err"]
    return box["out"]
  • The strategy runs in a daemon thread. Python can't safely interrupt a thread mid-CUDA-call, so we don't kill it — we just stop waiting for it.The thread continues in background but its result is ignored.
  • th.join(deadline) blocks the main thread for at most that many seconds.Only safe way to impose a time limit on GPU computation.
  • Daemon threads are killed when the main process exits.No zombie processes from hung GPU calls.

The Pacer — Rate Limiter

polimibot/runner.py — _Pacer
class _Pacer:
    """Ensures at least `min_delay` seconds between wait() calls."""
    def __init__(self, min_delay: float):
        self._min_delay = min_delay  # 1.5 seconds
        self._last = 0.0

    def wait(self):
        elapsed = time.monotonic() - self._last
        if 0 < elapsed < self._min_delay:
            time.sleep(self._min_delay - elapsed)
        self._last = time.monotonic()
  • time.monotonic() never goes backward (unlike wall-clock time adjusted by NTP).Correct even if system clock is adjusted.
  • Only sleeps for the remaining gap. If the LLM took 2.3s (longer than 1.5s min), no sleep needed.Never adds unnecessary delay.

The runner is the safety net. No strategy bug can crash a live game. Timeouts, exceptions, parse failures, CUDA errors — all are caught and handled by submitting the fallback answer. The game continues. All errors are logged.

Strategies

The Strategy Pattern — The Contract Everyone Signs

A game show host doesn't care how a contestant decides. Whether they're a genius, have a cheat-sheet, or flip a coin — the interaction is always: host asks → contestant picks A/B/C/D → host reveals. The Strategy pattern is that contract: a common interface with interchangeable implementations.

polimibot/strategies/base.py — The contract
@dataclass(frozen=True)
class StrategyInput:
    """Everything a strategy needs to answer one question."""
    question: str
    options: tuple[str, str, str, str]
    level: int                    # 1–15
    max_level: int = 15
    category: Optional[Category] = None
    competition_id: int = 0
    time_budget_seconds: float = 25.0

@dataclass(frozen=True)
class StrategyOutput:
    """What a strategy returns after deciding."""
    chosen_index: int             # 0=A, 1=B, 2=C, 3=D
    confidence: float = 0.25     # probability of correctness
    rationale: str = ""          # human-readable reasoning
    is_abstain: bool = False     # "I don't know" flag
    extras: dict = field(default_factory=dict)

class Strategy(ABC):
    """Abstract base class — all strategies implement this."""
    name: str = "strategy"

    def warm_up(self) -> None:
        """Load GPU models, build caches. Called once before game starts."""

    def shutdown(self) -> None:
        """Release GPU memory. Called after game ends."""

    @abstractmethod
    def answer(self, inp: StrategyInput) -> StrategyOutput:
        """Given a question, return a decision. Must be thread-safe."""
  • Both Input and Output are frozen dataclasses — immutable after creation. You cannot accidentally mutate them in an ensemble scenario where multiple strategies share the same input.Prevents corruption in concurrent/ensemble scenarios.
  • Default confidence is 0.25 (random chance for 4 options). Strategies that "know" they're right report higher confidence; the ensemble weights votes by this.Calibrated confidence enables intelligent ensemble fusion.
  • The abstention flag explicitly says "I couldn't answer." The runner then uses the fallback index. This is better than silently guessing "A".Honest reporting of uncertainty.
  • Lifecycle hooks allow strategies to load 7B models onto GPU once, then answer many questions without reloading.GPU model loading takes 10-30 seconds. warm_up() ensures this happens once, not per question.

RAGStrategy — Retrieval-Augmented Generation

The RAG strategy retrieves relevant Wikipedia passages before asking the LLM to answer. This turns a "closed-book exam" into an "open-book exam."

polimibot/strategies/rag_strategy.py — simplified
class RAGStrategy(Strategy):
    """Retrieve → Augment → Generate."""

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        # 1. Retrieve relevant passages
        hits = self.retriever.retrieve(
            query=inp.question, k=self.k,
            category=inp.category,
        )
        # 2. Format context
        context = self._format_passages(hits)
        # 3. Build RAG-augmented prompt
        messages = build_messages_with_context(
            inp.question, inp.options, context,
            category=inp.category, style=self.style,
        )
        # 4. Generate answer
        if self.use_score_options:
            return self._answer_via_logits(messages)
        return self._answer_via_generation(messages)

Typical accuracy: ~78% (+10% over baseline). The biggest improvement comes on History and Science categories where Wikipedia coverage is strong and questions ask about specific facts the LLM might not remember.

EnsembleStrategy — Strength in Numbers

Think of a jury trial. One person might be wrong. But when 12 people independently reach the same verdict, the probability of a collective error drops dramatically. The EnsembleStrategy is a jury of AI strategies — each votes independently, and the majority wins (weighted by confidence).

polimibot/strategies/ensemble_strategy.py — simplified
class EnsembleStrategy(Strategy):
    """Run multiple strategies, fuse their votes."""

    def __init__(self, strategies: list[Strategy], method="confidence_weighted"):
        self.strategies = strategies
        self.method = method

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        votes = []
        for s in self.strategies:
            out = s.answer(inp)
            if not out.is_abstain:
                votes.append(out)

        if not votes:
            return StrategyOutput(chosen_index=0, is_abstain=True)

        # Confidence-weighted fusion
        scores = [0.0] * 4
        for v in votes:
            scores[v.chosen_index] += v.confidence

        best = max(range(4), key=lambda i: scores[i])
        total = sum(scores)
        return StrategyOutput(
            chosen_index=best,
            confidence=scores[best] / total if total > 0 else 0.25,
        )
  • Each strategy votes independently. No strategy sees another's answer.Independence is crucial — correlated errors don't cancel out.
  • Strategies that abstain are excluded from voting. Their weight redistributes to confident strategies.A confused strategy shouldn't dilute a confident one.
  • Votes are weighted by confidence. A strategy saying "90% sure it's B" outweighs one saying "30% sure it's A."Well-calibrated confidence scores make the ensemble much more effective.

TieredStrategy — Difficulty-Based Routing

Like a hospital triage system. Easy cases (levels 1-5) go to the fast-track nurse (baseline LLM — cheap, quick). Medium cases (levels 6-9) go to the specialist (RAG — more thorough). Hard cases (levels 10-15) go to the full team consultation (ensemble — maximum accuracy). Maths always goes to the calculator (regardless of level).

polimibot/strategies/tiered_strategy.py
class TieredStrategy(Strategy):
    """Route questions by difficulty level."""

    def __init__(self, easy, medium, hard, *, maths_override=None,
                 easy_ceiling=5, hard_floor=10):
        self.easy = easy        # levels 1-5: cheap baseline
        self.medium = medium    # levels 6-9: RAG
        self.hard = hard        # levels 10-15: ensemble
        self.maths = maths_override  # ALL maths regardless of level

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        # Maths override: always use the calculator path
        if self.maths and inp.category == Category.MATHS:
            return self.maths.answer(inp)

        # Route by difficulty
        if inp.level <= self.easy_ceiling:
            return self.easy.answer(inp)
        elif inp.level >= self.hard_floor:
            return self.hard.answer(inp)
        else:
            return self.medium.answer(inp)

Why tier? Easy questions don't need the overhead of retrieval or ensemble voting — a baseline LLM handles "What color is the sky?" just fine. Hard questions at level 13+ have high stakes (€125K–€1M) and need every advantage. Tiering gives you ensemble-level accuracy where it matters while keeping latency low on easy questions.

Deployment Compositions — Confidence Gating & Tools

Three more strategies compose the basics for deployment, all sharing the same answer() contract. They exist to spend the expensive arms (retrieval, the LLM) only when they pay off.

StrategyRouting logicKey knobs
ConfidenceGatedStrategyRun a fast primary; escalate to a heavier fallback only when the primary's logit margin (top1−top2) is below threshold. Listed categories always escalate.margin_threshold=0.20, always_fallback_categories (e.g. NEWS)
ConfidenceToolStrategyPin a fast LLM answer; try deterministic tools only when LLM confidence is low. The pinned answer always ships — zero timeout risk.confidence_threshold=0.70, tool_time_budget=5.0

Why gate? A calibrated primary (measured ECE ≈ 0.04 on History) gains nothing from retrieval when it is already confident, and unauthenticated Wikipedia rate-limits at ~10 requests per burst. Routing the confident majority through the bare LLM and reserving retrieval for the low-margin tail keeps API volume under the burst threshold without moving the accuracy ceiling.

Prompt Engineering

The PromptStyle Enum — 5 Experiment Conditions

In PoliMillionaire, the same LLM answers trivia questions across six categories. The difference between 60% and 85% accuracy often isn't a better model — it's a better prompt.

Core Philosophy: A PromptStyle is an experiment condition, not a magic string. Only the templates.py module knows what text goes to the model. Everything else speaks in terms of the enum — making ablation studies trivial.

Prompt Engineering

System Prompts — Category-Specific Instructions

Each category gets a different system prompt that teaches the model the gotchas of that domain. These are instructive, not flattering — generic praise ("You are an expert") leaves accuracy on the table.

Design Insight: The system prompts warn about specific distractor patterns the trivia game uses. For History: "similar-sounding dynasties, namesake successors." For Science: "units confusion, necessary vs sufficient conditions." This primes the model to double-check exactly the things the distractors exploit.

system ENTERTAINMENT
You are answering multiple-choice trivia on film, music, television, and pop culture. Trivia distractors often pair real co-stars from the same era, films by the same director, or bands sharing members. Verify the specific year, director, album, or actor before committing.
system HISTORY
You are answering multiple-choice trivia on ancient history, classical civilisations, and political history. Watch for plausible distractors that confuse adjacent centuries, similar-sounding dynasties, or namesake successors (e.g. Caesar vs Augustus, Henry V vs Henry VIII). When the LLM generates free text, we need to extract the answer letter. The parser uses a carefully ordered cascade from most specific to least specific:

1

2

3

4

polimibot/prompts/templates.py — parse_answer()
def parse_answer(text: str) -> Optional[int]:
    """Extract 0-based option index from model output. None if unparseable."""
    if not text:
        return None
    for pattern in _STRUCTURED_PATTERNS:
        m = pattern.search(text)
        if m:
            return ord(m.group(1).upper()) - ord("A")
    # Bare letter — LAST match wins
    matches = _BARE_LETTER.findall(text)
    if matches:
        return ord(matches[-1].upper()) - ord("A")
    return None  # unparseable → strategy abstains

On Parse Failure: When parse_answer() returns None, the strategy marks the response as is_abstain=True. The runner submits the fallback. The ensemble redistributes weight to surviving strategies.

def score_options(self, messages, letters=("A", "B", "C", "D")):
    """One forward pass → probabilities over answer letters."""
    prompt = self._apply_template(messages, add_generation_prompt=True)
    prompt = prompt + "Answer: "  # model completes THIS

    inputs = self._tokenizer(prompt, return_tensors="pt").to(self._model.device)
    with torch.inference_mode():
        last_logits = self._model(**inputs).logits[0, -1, :]

    # Softmax over just the 4 letter token IDs
    raw = torch.tensor([last_logits[tid] for tid in letter_ids.values()])

What is RAG? — The Core Idea

Imagine a quiz show contestant allowed to bring a mini-encyclopedia. Before answering, they flip to relevant pages and read the key facts. RAG (Retrieval-Augmented Generation) gives the LLM this same power: instead of answering purely from memory (training data), it first looks up relevant Wikipedia passages, then answers with that evidence in context.

Without RAG: The LLM answers from training data alone — which may be outdated, incomplete, or wrong for obscure facts.

With RAG: The LLM gets relevant passages injected into its prompt. It becomes an "open-book exam."

FAISS Semantic Search — Finding Meaning

Think of a library where books are shelved by topic similarity, not alphabetically. A book about "Roman military campaigns" sits next to "Caesar's conquests" even though their titles don't share words. FAISS (Facebook AI Similarity Search) does this with vectors: documents are encoded as high-dimensional vectors, and similar meanings cluster together in vector space.

The semantic search pipeline:

  1. Offline: Every Wikipedia chunk is embedded into a 768-dim vector using a sentence transformer model (BAAI/bge-base-en-v1.5)
  2. Stored: All vectors are indexed in a FAISS flat index (exact nearest-neighbor search)
  3. At query time: The question is embedded with the same model → find the k nearest vectors → retrieve those chunks

Why sentence transformers? Traditional word embeddings (Word2Vec) can't capture full sentence meaning. Sentence transformers are trained specifically on semantic similarity tasks: "capital of Italy" and "Rome is the seat of Italian government" get similar vectors even though they share few words.

Hybrid Retrieval & RRF Fusion

Like asking two librarians with different specialties. One organizes by topic (FAISS), the other by keywords (BM25). Each produces a ranked list. Reciprocal Rank Fusion (RRF) merges them: a document ranked #1 by either librarian gets a high score; a document ranked #1 by BOTH gets the highest score.

polimibot/rag/fusion.py — RRF algorithm
def reciprocal_rank_fusion(
    ranked_lists: list[list[tuple[Chunk, float]]],
    k: int = 60
) -> list[tuple[Chunk, float]]:
    """Merge multiple ranked lists using RRF. k=60 is standard."""
    scores: dict[str, float] = defaultdict(float)
    chunk_map: dict[str, Chunk] = {}

    for ranked_list in ranked_lists:
        for rank, (chunk, _original_score) in enumerate(ranked_list, start=1):
            key = chunk.unique_id
            scores[key] += 1.0 / (k + rank)  # RRF formula
            chunk_map[key] = chunk

    # Sort by fused score descending
    sorted_items = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [(chunk_map[key], score) for key, score in sorted_items]
  • The constant k=60 is standard in information retrieval literature. It dampens the difference between rank 1 and rank 2 while still respecting rank order.Prevents a single high-ranked result from dominating. Ensures both retrieval methods have influence.
  • Scores are additive across lists. A chunk ranked #1 by FAISS (score 1/61) and #3 by BM25 (score 1/63) gets total score 1/61 + 1/63.Documents found by BOTH methods get a natural boost.

Cross-Encoder Reranking — The Quality Filter

Like a second-round interview. The first round (FAISS + BM25) casts a wide net. The second round (cross-encoder) is a more expensive but more accurate evaluation: it reads the question AND each passage together, scoring relevance with full attention. Only the top passages survive.

The cross-encoder sees both question and passage as a single input, allowing it to model fine-grained interactions between them. This is much more accurate than bi-encoder similarity (which encodes them separately) but also much slower — hence it's only applied to the top candidates.

Retrieve-then-rerank pattern: Retrieve 20 candidates cheaply (FAISS+BM25), rerank to find the best 3-5. This gives you cross-encoder quality at bi-encoder speed.

IndexGrower — Live Learning During Games

Like a student adding flashcards during the exam. When the bot encounters a question the category-graph-seeded offline index can't answer, it fetches the relevant Wikipedia article right then. If the server confirms the answer was correct, that article gets permanently added to the knowledge base. Wrong answers are discarded — never pollute the index with misinformation.

class IndexGrower:
    """Grows the RAG index during live games."""

    def fetch_and_stage(self, query: str) -> list[Chunk]:
        """Fetch Wikipedia article, chunk it, stage for confirmation."""
        article = self._wiki_api.search(query)
        chunks = self._chunker.chunk(article)
        self._staged[query_id] = chunks
        return chunks

    def confirm(self, query_id: str):
        """Answer was correct → add staged chunks to permanent index."""
        chunks = self._staged.pop(query_id, [])
        self._index.add(chunks)  # permanent!

    def discard(self, query_id: str):
        """Answer was wrong → throw away staged chunks."""
        self._staged.pop(query_id, None)  # never enters the index

    def flush(self):
        """Persist the grown index to disk."""
        self._index.save()

The Tool Interface — Two Methods, That's It

Think of each tool as a specialist doctor in an ER. The triage nurse (ToolStrategy) shows each patient's chart to specialists in order. The specialist either says "This is my case" (can_handle → True, use → StrategyOutput) or "Not my specialty, pass" (can_handle → False or use → None).

polimibot/tools/base.py — The Tool contract
class Tool(ABC):
    """A tool either answers confidently or abstains (returns None)."""
    name: str = "tool"

    @abstractmethod
    def can_handle(self, inp: StrategyInput) -> bool:
        """Fast guard based on metadata (category, level).
        No computation here — must be instant."""

    @abstractmethod
    def use(self, inp: StrategyInput) -> Optional[StrategyOutput]:
        """Attempt to answer. Return None to abstain."""

Two levels of filtering: can_handle filters by question TYPE (fast, zero-cost). use filters by question CONTENT (may involve parsing). This two-stage design prevents expensive parsing on questions that obviously aren't relevant.

The Safe Evaluator — A Sandboxed Calculator

Imagine building a calculator for a children's classroom. You want them to compute 2+2 and sqrt(16), but you absolutely do NOT want them to type import os; os.system("format C:"). The safe evaluator is that childproof calculator — it only allows whitelisted math operations and blocks everything else at the AST level.

Python's eval() is a remote code execution vulnerability. The LLM's output feeds into this evaluator. If you use raw eval(), a malformed output like __import__('os').system('rm -rf /') would execute arbitrary system commands. The AST sandbox makes this physically impossible.

polimibot/tools/calculator.py — _eval_node()
def _eval_node(node):
    if isinstance(node, ast.Constant):
        if isinstance(node.value, (int, float)):
            return node.value
        raise ValueError(f"Non-numeric constant: {node.value!r}")

    if isinstance(node, ast.BinOp):
        fn = _BINARY_OPS.get(type(node.op))
        if fn is None:
            raise ValueError(f"Operator not allowed")
        return fn(_eval_node(node.left), _eval_node(node.right))

    if isinstance(node, ast.Call):
        if not isinstance(node.func, ast.Name):
            raise ValueError("Only simple function names allowed")
        fn = _ALLOWED.get(node.func.id)
        if fn is None:
            raise ValueError(f"Function '{node.func.id}' not in whitelist")
        return fn(*[_eval_node(a) for a in node.args])

    # ANYTHING ELSE → REJECTED (deny by default)
    raise ValueError(f"AST node {type(node).__name__} not allowed")

polimibot/tools/maths_tool.py — MathsTool class
class MathsTool(Tool):
    """Deterministic arithmetic solver. Only answers when it can prove it."""
    name = "maths_tool"

    def can_handle(self, inp: StrategyInput) -> bool:
        return inp.category == Category.MATHS  # instant check

    def use(self, inp: StrategyInput) -> Optional[StrategyOutput]:
        expr = _extract_expression(inp.question)
        if expr is None:
            return None          # can't parse → abstain

        try:
            result = safe_eval(expr)
        except Exception:
            return None          # math error → abstain

        if not isinstance(result, (int, float)) or not math.isfinite(result):
            return None          # NaN/Inf → abstain

        idx = _match_options(float(result), inp.options)
        if idx is None:
            return None          # doesn't match any option → abstain

        return StrategyOutput(
            chosen_index=idx, confidence=0.99,
            rationale=f"MathsTool: {expr} = {result}",
        )

Count the exit points: There are 4 places where the tool returns None (abstains). Only ONE path leads to an answer. This is intentional — narrow corridor of success, wide highway of graceful failure. When it DOES answer, you can trust it.

Symbolic & Statistical Tools — Beyond Arithmetic

MathsTool is a pocket calculator; some Maths questions need a graphing calculator and a stats table. "15% of 400?" is arithmetic. Three specialists, one chain. ToolStrategy — Chain of Responsibility

Like a customer service escalation chain. Tier 1 (tool) tries to solve the problem. If they can't → escalate to Tier 2 (next tool). If nobody can → escalate to the manager (fallback LLM). Adding a new specialist requires ZERO changes to existing code.

polimibot/strategies/tool_strategy.py
class ToolStrategy(Strategy):
    """Try registered tools in order; fall back to an LLM strategy."""

    def __init__(self, tools: list[Tool], fallback: Strategy):
        self.tools = tools
        self.fallback = fallback

    def answer(self, inp: StrategyInput) -> StrategyOutput:
        for tool in self.tools:
            if tool.can_handle(inp):       # fast gate
                out = tool.use(inp)        # attempt
                if out is not None:
                    return out             # tool answered!
        # All tools abstained
        return self.fallback.answer(inp)   # LLM takes over
  • Tools are tried in insertion order. Put the most confident/fastest tool first.Priority chain: cheapest answer wins.
  • First can_handle (free), then use (may involve parsing). If can_handle is False, use() is never called.Don't waste time parsing a history question as math.
  • The fallback is itself a Strategy — could be RAG, baseline, or even another ToolStrategy. Arbitrary nesting.Composability: ToolStrategy(tools=[MathsTool()], fallback=RAGStrategy(...)) combines both.

def answer(self, inp: StrategyInput) -> StrategyOutput:
    deadline = time.monotonic() + inp.time_budget_seconds - 1.0
    messages = self._initial_messages(inp)

    for iteration in range(self.max_iterations + 1):
        if time.monotonic() >= deadline:
            break  # out of time!

        resp = self.llm.generate(messages, max_new_tokens=200, temperature=0.0)
        text = resp.text.strip()
        messages.append({"role": "assistant", "content": text})

        # Branch 1: Tool call detected
        call = _extract_call(text)
        if call and time.monotonic() < deadline:
            tool_name, expression = call
            observation = _run_calc(expression)  # safe_eval
            messages.append({"role": "user",
                "content": f"Tool result: {observation}\nNow give your final answer."})
            continue  # next iteration

        # Branch 2: Final answer detected
        idx = parse_answer(text)
        if idx is not None:
            return StrategyOutput(chosen_index=idx, confidence=0.70)

        # Branch 3: Nudge the model
        messages.append({"role": "user",
            "content": "Please write your final answer: 'Answer: '."})

    # Exhausted: salvage from trace
    return StrategyOutput(chosen_index=0, is_abstain=True)

Security Design — Defense in Depth

Think of airport security with multiple checkpoints. Even if a passenger (LLM output) gets through one check, there are more layers behind it. The AST parser is the X-ray machine. The whitelist is the no-fly list. The deny-by-default fallthrough is the armed guard at the gate.

graph TD INPUT["LLM Output (untrusted)"] --> L1["Layer 1: _extract_call — regex + paren scan"] L1 --> L2["Layer 2: ast.parse(mode='eval') — rejects statements"] L2 --> L3["Layer 3: _eval_node whitelist — only safe ops/funcs"] L3 --> L4["Layer 4: Deny-by-default fallthrough"] L4 --> L5["Layer 5: Exception handler — any error → abstain"] L5 --> SAFE["Safe numeric result"]

Five layers of security — each catches threats the previous might miss

Key principle: deny by default. The _ALLOWED whitelist is finite and small — exactly 15 functions plus the constants pi and e, over 7 binary and 2 unary operators. Rather than trying to blacklist dangerous functions (you'll miss some), we whitelist only the safe ones. Everything else is rejected by the fallthrough. The agent's sympy_solve path adds its own deny-list (_BLOCKED: dunders, import, open/exec/eval/getattr) plus a SymPy-Basic result check, because sympify() uses eval() internally.

The gold set only needs a single pass (every correct=True record is trivially a gold item). The wrong set needs two passes, because we want to also recover the correct answer when possible — so the item is maximally useful for training.

Calibration — "Am I as Sure as I Think?"

Imagine a weather forecaster. When they say "90% chance of rain," it should actually rain 9 out of 10 times. If it only rains 6 out of 10 → they're overconfident. Expected Calibration Error (ECE) measures this gap between stated confidence and actual accuracy.

def compute_calibration(confidences, corrects, n_bins=10):

        os.fsync(self._fh.fileno())  # crash-safety: force to disk
        self._fh.close()

    def _write(self, rec):
        self._fh.write(json.dumps(asdict(rec)) + "\n")
        self._fh.flush()  # tail -f friendly
  • os.fsync() forces data to disk on close. Even if the process crashes, committed data is safe.Crash safety: mid-game crash doesn't lose questions already played.
  • Each write calls flush() immediately. You can tail -f the file in real-time.Live observability: watch the bot play from another terminal.
  • JSONL is append-only, human-readable, streamable, and crash-safe. Each line is independent.For experimental logging, it's the sweet spot vs CSV or SQLite.

Timestamped Outputs — Never Overwrite a Result

Like dated lab notebook pages. Every eval run, gold-set build, calibration plot, and tier sweep now stamps a UTC timestamp into its output filename. Running the same script twice produces two files — not one silently overwritten.

def ts() -> str:
    """Return a filesystem-safe UTC timestamp string: YYYYMMDD_HHMMSS.

    Use this to stamp output filenames so each run creates a new file
    and never silently overwrites previous results.

    Example:
        from polimibot.config import PATHS, ts
        out = PATHS.eval_dir / f"report_rag_{ts()}.json"
    """
    return _dt.datetime.now(_dt.timezone.utc).strftime("%Y%m%d_%H%M%S")

def save_report(report: EvalReport, name: str, eval_dir: Path) -> Path:
    """Serialise report to eval_dir/{name}_{ts()}.json.

    A UTC timestamp is embedded in the filename so each run produces a
    new file and no previous results are silently overwritten.
    """
    eval_dir.mkdir(parents=True, exist_ok=True)
    out = eval_dir / f"{name}_{ts()}.json"   # ← stamped
    report.save(out)
    return out
  • UTC, not local time — sweeps on different machines sort consistently.Avoids the daylight-saving "two files with the same timestamp" trap.
  • Format YYYYMMDD_HHMMSS — sortable lexicographically.ls data/eval/ orders runs chronologically without extra parsing.
  • Scripts that take --out still respect an explicit override; the timestamp is only the default.Reproducibility scripts can pin a known path when needed.

The leaderboard scanner reads every JSON in data/eval/. Timestamped files let you keep historical runs alongside new ones — the leaderboard naturally shows progress over time, not just the latest result.
